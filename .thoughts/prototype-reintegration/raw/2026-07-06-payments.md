## 0. STRANGER TEST — status first (rule 15)

**Current status: CANNOT PASS — by design intent it will, but every external prerequisite is still unprovisioned and the live settlement spike is unproven.**

The stranger loop this domain must close: a brand-new buyer (fresh email, funds held ONLY on a non-Arbitrum chain, e.g. USDC on Base) lands on a merchant page whose merchant signed up through Tab's own real signup → clicks `Pay $X` → enters email + real Magic OTP → sees their real unified balance → pays → merchant's Arbitrum One address receives USDC (visible on Arbiscan) → merchant's webhook endpoint receives a signed POST. No seeded state, no demo account, no pinned tenant anywhere in this design. Value exits Tab's rail at the moment of settlement (non-custodial — see §7 money loop).

What blocks the stranger today: Particle project keys, Magic app keys + allowed origins, a funded buyer wallet, a deployed HTTPS origin, a database — all named in §8 Blockers with Abu-actions — plus the **unproven external-receiver live spike (spec OQ-5 / Task #2)**, which gates the single most important row in this matrix (row B4-4).

## 1. Verified capability findings from this pass (new evidence, changes labels)

These were open questions in discovery; they are now verified against the cloned docs / published packages:

1. **Magic white-label email OTP is REAL and documented.** `loginWithEmailOTP({ email, showUI: false })` — "When `false`, you can implement a custom UI to continue the email OTP flow" (`Reference/magic-docs/embedded-wallets/authentication/login/email-otp.mdx:18`). Full event flow documented: receive `EmailOTPSent` / `InvalidEmailOtp` / `ExpiredEmailOtp`, emit `VerifyEmailOtp(otp)` / `Cancel` (`.../sdk/client-side/javascript.mdx:151-247`, `.../sdk/resources/types.mdx:188-232`). Must pass `deviceCheckUI: false` alongside if Device Verification is dashboard-enabled (`javascript.mdx:117`). **→ The prototype's #1 tension (custom Tab-styled B2/B3 inside one sheet vs spec `showUI: true`) is RESOLVED in favor of the visual law: buildable for real. Spec delta D1.**
   - NOT provided by the SDK: a resend event (emit enum has only `VerifyEmailOtp`/`Cancel`, `types.mdx:223-226`) and a documented rate-limit event/lockout duration. See rows B3-6/B3-5.
2. **magic-sdk ≥30 removed flat `publicAddress` from `getInfo()`** — address now at `wallets.ethereum.publicAddress` (`javascript.mdx:915,938`). Spec R-TAB-4's "getInfo() yields publicAddress" is stale at magic-sdk@33.9.0. Delta D2.
3. **Magic client embedded wallet CAN sign EIP-7702 authorizations in-browser**: `magic.wallet.sign7702Authorization({ contractAddress, chainId, nonce? }) → { contractAddress, chainId, nonce, v, r, s }`, available since magic-sdk@33.4.0 (`javascript.mdx:757-777`). Maps onto Particle's `userOp.eip7702Auth { chainId, nonce, address }` (SDK 2.0.3 `dist/index.d.ts:237-243`); our adapter serializes `(v,r,s)` into Particle's `EIP7702Authorization { userOpHash, signature }` (d.ts:347-350). **Adapter serialization is our glue — exact format verified at the live spike.** `magic.rpcProvider` is the EIP-1193 provider for the `personal_sign` of `rootHash` (`javascript.mdx:1674`).
4. **`magic.user.isLoggedIn()` documented** (`javascript.mdx:1021-1042`) — the real source for RETURNING_USER_SKIP.
5. **UA SDK 2.0.3 pinned facts** (from `.thoughts/research/2026-07-06-eco-particle-magic.md`): `createTransferTransaction({token{chainId,address}, amount, receiver})` intact (d.ts:362, :72-76); `sendTransaction(tx, signature, authorizations?)` 3-arg intact (d.ts:366); nested `smartAccountOptions.useEIP7702: true` is the documented init; **`tradeConfig.universalGas` REMOVED (2.0.2)** and **flat `ownerAddress` REMOVED (2.0.3)** — spec R-TAB-5 snippet must be corrected (delta D3); `UNIVERSAL_ACCOUNT_VERSION = "2.0.1"`, all accounts treated as V2 → **the UA-V1 migration interstitial question (story-2 OQ3 / cover OQ-B2) is dead** (delta D20).
6. **USDC on Arbitrum One = `0xaf88d065e77c8cC2239327C5EDb3A432268e5831`** — triple-confirmed native USDC w/ EIP-3009, 6 decimals (`.thoughts/research/2026-07-06-eco-x402-mcp.md`), and USDC is a UA-V2 primary token on Arbitrum (`chains.mdx` availability table). **Spec OQ-4 (USDT vs USDC) resolved: USDC.** Delta D5.

## 6. Webhook contract (concretized, D8)

```
POST {merchant webhook URL}
Content-Type: application/json
X-Tab-Signature: t=<unix-seconds>,v1=<hex hmac-sha256(secret, `${t}.${rawBody}`)>

{ "transactionId": "0x…", "tokenChanges": [ { "token": "0xaf88…5831", "chainId": 42161, "amount": "12000000", "receiver": "0x…" } ] }
```
- Fires only after chain-verified settlement (API-4). Target: first attempt ≤15s after `sendTransaction` resolve under normal RPC conditions (answers story-1 OQ1).
- Per-request timeout 10s; non-2xx or timeout → retry at +1m, +4m, +16m; then terminal `gave up` (answers story-1 OQ2; matches drawn "attempt N of 3" / "Next retry in 4 min").
- Merchant verifies with the shown-once `whsec_` secret (merchant-domain M5 owns the config UI).

## 7. Money loop (rule 10 — answered before Abu asks)

- **Merchant:** settlement is direct on-chain to the merchant's OWN `receiving_address` on Arbitrum One. Tab is non-custodial — no balance ever sits with Tab, so there is nothing to "withdraw from Tab"; the merchant's USDC is theirs at settlement instant (visible on Arbiscan).
- **Buyer:** the buyer's funds live in their own Magic-EOA/UA (they hold the key via Magic). Paying IS value leaving. Any residual balance remains fully under the buyer's control (usable in any UA-compatible surface, e.g. universalx.app); Tab never custodies it. Buyer funding (how money got in) is the buyer's own wallet history — Tab adds no deposit surface in this domain.
- **Refunds:** no refund button exists in the prototype and none is faked. Roadmap statement only (merchant sends USDC back on their own) — REAL_LATER, never a dead button.
