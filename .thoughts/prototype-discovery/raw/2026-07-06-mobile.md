## HIDDEN REQUIREMENTS (the UI now implies; nobody specced)

1. **Float-aggregate endpoint:** big number = sum of on-chain float balances across Base/Arbitrum/Polygon; backend must read 3 chains and aggregate (and decide whether UA treasury total appears on mobile at all).
2. **Notification inbox or badge semantics:** bell badge (2/3) implies unread-notification state + read-marking API; either design an inbox screen or drop the badge.
3. **Cycle indexing:** receipt cycle line needs per-receipt `index / total-in-cycle` + cumulative-at-view (`Payment 12 of 23 · cumulative $4.20 / $10.00`) from the ledger.
4. **Next-reset timestamp:** "Resets daily · next Jul 3, 00:00 UTC" → cap config must expose computed next reset (UTC-midnight anchored).
5. **Blocked-event target network:** blocked receipts render "Base (target)" `eip155:8453` → the cap-gate 403 path must persist the would-be settlement network parsed from the 402 `accepts[]` even though nothing was submitted.
6. **Explorer URL mapping per chain:** txHash → Basescan/Arbiscan/Polygonscan URL builder keyed by CAIP-2.
7. **Lock-screen Revoke:** service worker must handle a notification action by firing an authenticated HTTP soft-pause — requires a client-persisted auth credential usable from SW context, plus idempotency; iOS web push lacks action buttons (platform risk, unverified for current iOS).
8. **P5 detail rows can't ship as-is:** Web Push carries plain text; detail rows must be flattened into the body string; per-tier icon color/border not OS-controllable — in-app rendering only.
9. **Offline machinery:** "last seen 14:31" + STALE pill + dimmed controls → client caches last successful fetch timestamp + payload; online/offline listener; controls hard-disabled offline.
10. **Confirm-string source:** L4 type-to-confirm uses the agent display name → agent must have a canonical, user-known name; hold-to-delete needs a 3s press timer with cancel-on-release.
11. **Deep-link routing:** P4/P5 taps land on Overview / Receipt-by-id / Revoke sheet states → PWA needs stable URL scheme (ties to OQ-P5 Next.js-route vs separate build).
12. **Optimistic status sync:** success toast claims "status chip updated" → revoke response must return the new agent status; Overview state must update without full reload (3s polling per resolved OQ-W6/P2 is the floor).
13. **Single-agent scoping:** header hardcodes one agent, no switcher — PWA is per-agent; account→agent resolution must be decided.
14. **Controls tab = sheet, not page:** tab bar's third tab opens an overlay sheet over the current screen — nav framework must support that pattern.
15. **Failed-with-txHash rendering:** prototype only shows hash-less failures; an on-chain revert (hash exists, settlement failed) has no design — implementation must degrade gracefully (show hash + Failed pill).

4. Keep the L1/L2 confirm dialogs the footer invented, or restore surface-map one-tap L1/L2 (confirmation only for L3/L4)? Bears directly on the "one-tap revoke" pitch.
5. Is lock-screen Revoke (P5 Tier-3 action) accepted as Android-only, given iOS Safari web push has no action buttons? (unverified against current iOS)
6. P2 blocked rows hide network while P3 blocked shows "(target)" — intentional asymmetry?
7. `halted_readonly` feed banner copy is paused-specific — need per-status variants (Frozen/Cancelled/Nuclear)?
8. Mobile "rebalance in progress" visibility: reinstate the dropped "Topping up…" badge (it's the cross-chain flagship) or keep mobile silent per OQ-W9?
9. P6 compact chip "Cancelled" vs P1 "Cancelled — provisioning required": which exact string is canonical for the §945 consistency rule?
10. Tier-2 push copy: canonical surface-map string or the designer's shortened form?
