## Reading (descriptive of what the docs support)

**Strongest for headless / agent use.** Three SDKs document a first-party headless path with explicit agent framing. **ZeroDev** is the most explicit: its page calls the AI-agent wallet "yes / first-class," with session-key permissions and an official Coinbase AgentKit integration plus a LangChain quickstart, and its CLI/server/headless support extends to Go/Python/Rust/C via the Omni SDK. **Openfort** documents a dedicated server SDK (`@openfort/openfort-node`), self-hostable key management (Shield/OpenSigner), a CLI repo, and a documented session-keys-for-agents pattern (its agent-wallets cross-reference calls it "the cleanest off-the-shelf agent-wallet kit among the sponsors"). **Magic** has purpose-built headless **Server Wallets** (TEE keys, REST, "ideal for backend systems and headless applications") — but its page rates the AI-agent-wallet column only "partial" because there is no dedicated agent product/branding. **Particle UA** also documents server-side/headless 7702 signing, but its agent-wallet support is "partial / build-it-yourself." **Arbitrum** offers full headless backend/settlement support but is a chain, not an agent-wallet provider (agent column "partial," it settles agent txs built on the others).

**Strongest for mobile.** **Openfort** and **Magic** both document the broadest first-party mobile coverage: React Native packages plus native iOS/Android (and Openfort adds Unity/Unreal; Magic adds Flutter). **ZeroDev** is "partial → yes" — React Native works via community Expo starters rather than a first-party RN package, with native Omni SDK bindings for iOS/Android. **Arbitrum** is "yes via standard EVM tooling" with no first-party mobile SDK. **Particle UA** mobile is "unknown / partial" — native/RN SDKs exist for Auth/Connect but no documented native UA-7702 path.

**Strongest for browser extension.** Only **Arbitrum** documents extension support, and that is because it is a chain that any injected EVM wallet (MetaMask, OKX) connects to via a manual network add — not because it ships an extension. None of the four wallet SDKs document a browser-extension wallet: **Openfort** is an explicit "no" (it is the in-app alternative to extension wallets, though it can link external EOAs via connectors), **Particle UA** is "no" for 7702 mode (JSON-RPC extension wallets can't sign Type-4 authorizations) and only works with them in Smart Account mode, while **ZeroDev** ("unknown") and **Magic** ("unknown / likely no") have no extension-specific documentation.

## Unknowns / to verify

- **Particle UA mobile/native:** whether a native (iOS/Android/RN) UA-7702 path exists — page marks it "unknown / partial," feasible via RN JS runtime but unverified for native [particle p.109].
- **Particle UA AI-agent wallet:** whether the server-side signing pattern is an officially sanctioned agent use case (page calls it build-it-yourself, "unverified as an official use case") [particle p.111]. The concept page additionally flags "Universal Agent Accounts" as possibly marketing/roadmap, not confirmed in cloned docs [agent-wallets, Open questions].
- **ZeroDev browser extension:** no extension-specific docs; whether the viem-based lib actually runs in an extension/service-worker JS context is unverified [zerodev p.105].
- **ZeroDev mobile:** RN is community-template-driven, not a first-party package; Omni SDK maturity for non-TS languages (Go/Python/Rust/C) is "newer than the TS SDK … depth/stability unverified" [zerodev p.106, p.117].
- **Openfort AI-agent wallet:** no single dedicated agent SDK; the agent pattern is assembled from session keys + backend wallets — extent of turnkey support is the "partial/yes" ambiguity [openfort p.102].
- **Magic browser extension:** "unknown / likely no (unverified)" — no official extension guide; iframe + OAuth-redirect model not documented for extension/service-worker contexts [magic p.98].
- **Magic AI-agent wallet:** "partial" with no dedicated AI-agent product; AI suitability is "unverified beyond recipes" (server wallets + x402 recipe + UOA) [magic p.101]. Concept page flags that the Magic $500 bonus is scoped to the *embedded* wallet, not headless server wallets — verify which product qualifies [agent-wallets, Open questions].
- **Arbitrum mobile:** "yes via standard EVM tooling," but there is no first-party Arbitrum mobile SDK — mobile support is inherited from EVM wallet tooling, not Arbitrum-specific [arbitrum p.75].
- **Arbitrum AI-agent wallet:** "partial" — no Arbitrum-native agent-wallet product; agent UX depends on the other SDKs' AA stacks settling on Arbitrum [arbitrum p.77].
- **Cross-cutting (from `../concepts/agent-wallets.md` Open questions):** whether UA 7702 mode supports a fully agent-held key and whether judges accept an agent (not a human) as the EOA owner; whether Particle's Biconomy-v2.0.0-only session-key limitation blocks combining session keys with UA 7702 mode; whether an agent can pay via x402 out of a chain-abstracted UA balance; exact status of agent standards (ERC-8004 mainnet date, ERC-7715/7710) — all pulled from secondary sources, verify against primary docs.
