# Prototype Discovery — Buyer checkout + SDK surface (raw findings)

- Date: 2026-07-06
- Inspector scope: `00 Cover.dc.html`, `01 Buyer Checkout.dc.html`, `_Buyer Frame.dc.html`, `_Checkout Sheet.dc.html`, `_Pay Button.dc.html`, `_Tab Auth.dc.html`, `_Tab Demo Page.dc.html`

- Prototype format: design-tool HTML exports (`support.js` runtime, `<x-dc>` markup, `sc-if`/`sc-for` conditionals, `dc-import` composition, JSON prop panel). Each `_*.dc.html` is a stateful component driven by an enum `state`/`view` prop; the numbered pages are static canvases that instantiate every state as a labeled frame. **The state enums map ~1:1 onto the product state machines — extremely useful for implementation, but everything is mocked: no inputs, no timers, no network.**

## VERDICT SUMMARY

- **State coverage: 25/25 buyer frames — NOTHING DROPPED.** Every state the surface map requires for B1 (7), B2 (2+back), B3 (7), B4 (4), B5 (1), B6 (4) is framed. M1 demo page carries all 8 surface-map states as enum options. MA1/MA2 auth states are all present in `_Tab Auth`.
- **Ban-list scan: PASS on buyer-facing strings.** No occurrence of chain / gas / bridge / Arbitrum / EOA / wallet / sign / EIP / token / USDC in `_Pay Button`, `_Checkout Sheet`, or `_Buyer Frame` buyer copy. One edge: the demo page's post-success dev annotation `wired via onSuccess(transactionId, tokenChanges)` (contains "token" inside `tokenChanges`) is rendered on the buyer-visible merchant demo page — see Deltas.
- **Biggest reintegration tension:** B2/B3 (Magic email + OTP) are drawn as fully custom Tab-styled views *inside Tab's own sheet* ("MAGIC SDK (MOCKED)" per the section labels; cover locks it as "Magic OTP + Tab checkout share one visual container"). Spec R-TAB-4 pins `magic.auth.loginWithEmailOTP({ showUI: true })` and the surface map's DECIDED table says "Magic SDK renders email + OTP screens; Tab does NOT build a custom OTP screen." The drawn design is only implementable with Magic headless mode (`showUI: false` + custom OTP UI via the SDK's event flow) — (unverified; must be checked against `.thoughts/wiki/sdks/magic.md` / Magic docs before planning). Visual law and spec are in direct conflict here.

## Hidden requirements the design reveals

1. **Magic headless OTP** (`showUI: false` + custom OTP UI via SDK events) is required to build the one-container sheet AND the merchant auth card as drawn — including resend-cooldown timer and red-box error states rendered by Tab. Conflicts with spec R-TAB-4's `showUI: true`. (Capability unverified — check `.thoughts/wiki/sdks/magic.md` / Magic docs.)
2. **Buyer reference-code scheme**: generate/store a short reference (e.g. `TAB-7B4A2`) mapped to `transactionId`, shown with copy-to-clipboard. Nobody specced this generator.
3. **Buyer receipt/confirmation email** from Tab ("A confirmation has been sent to …") — real email-sending capability + template, or the copy must go.
4. **PayButton variable-height contract**: ERROR and DISABLED_INSUFFICIENT_FUNDS render helper text below the 48px button (~76px total). SDK docs/layout must account for it.
5. **Intent fetch at mount**: demo page renders "Pay $12.00" in IDLE and has INITIALIZING/INTENT_ERROR — SDK must fetch the intent on mount (then re-validate at click per B1 LOADING). Amount formatting from decimal string `"5.00"`-style + currency → `$12.00` display.
6. **Responsive container switch**: same sheet renders as bottom sheet at mobile widths and centered 410px modal on desktop — SDK must implement both containers.
7. **Pre-sheet insufficient-funds knowledge**: B1 DISABLED_INSUFFICIENT_FUNDS appears on the bare merchant page (no sheet). Reachable only after B4 detected the shortfall (per map) — meaning the button must persist the insufficient state after sheet close; or, for returning sessions, the SDK could pre-check balance.
