## VERDICT ON THE TWO CRITICAL CHECKS

1. **Model picker: ABSENT everywhere. PASS.** The index explicitly labels W7 "CONNECT — REPLACES THE MODEL PICKER"; the Connect screen carries the read-only note "Leash has no model and makes no LLM calls. It only pays x402 requests your agent makes — and only within your cap." No model/LLM language appears in any Leash component. Matches corrected architecture 2026-07-02.
2. **Multi-chain floats: PRESENT. PASS (with one conceptual delta).** Base (PRIMARY badge) / Arbitrum / Polygon per-chain rows in W8; per-chain breakdown line on W1 ("Base 14.20 · Arb 6.10 · Pol 4.50"); NETWORK column on every W2 feed row; CAIP-2 network ids (eip155:8453 / 42161 / 137) + per-network explorer (Basescan / Arbiscan / Polygonscan) in W3. **Delta:** the owner's unified UA treasury total (`getPrimaryAssets().totalAmountInUSD`) is never shown as a number anywhere — see Delta D1.

## STRUCTURE OF THE HANDBACK

- `03 Leash Dashboard.dc.html` is a canvas page that imports each component at 1440×900 per state via `<dc-import name="..." state="...">`.
- Each `_Leash *.dc.html` is a self-contained component: static HTML template with `{{ }}` bindings + a `DCLogic` script that maps a `state` enum to mock data. **All data is mock data hardcoded in `renderVals()`** — this is the reintegration register's input.
- Designer split surface-map W7 (Agent Connection + Multi-Chain Float) into **two screens**: W7 "Connect your agent" (`_Leash Connect`) and W8 "Setup & funds" (`_Leash Setup`). Sidebar nav (identical across all screens, 7 items): Overview / Payments / Notifications (badge) / Cap & limits / Connect agent / Revocation / Setup & funds. Account row pinned bottom: "Abu / abu@example.dev".
- Visual register (per index copy): "Mercury/Ramp operator control surface"; spend bar green→amber→red is the primary signal; Blocked is a slate/ink policy chip, not an error red. Fonts Geist / Geist Mono; green #206B4E primary; amber #E0A33E warn; red #C13A40 danger; frozen steel-blue #3A6EA5 (deliberately distinct from Paused gray).

## HIDDEN REQUIREMENTS the design reveals (nobody specced, UI now implies)

- **H1:** Backend must serve a connected-client identity: "Claude Code" + transport + first-seen + per-cycle payment count. For MCP this presumably comes from the `initialize` handshake `clientInfo` (unverified) — the Leash MCP proxy must report client metadata to the control plane.
- **H2:** Auth-failure telemetry: "401 invalid key at 14:31" + REJECTED badge ⇒ the signer/API must log last auth failure (status + timestamp) per key and expose it to the dashboard.
- **H3:** `trigger_url` per receipt ("https://research.lab/tasks/lit-review-142") ⇒ the MCP proxy must capture what agent task initiated the paid fetch. For a stdio proxy the "trigger" is ambiguous — needs a real definition (tool-call context? referer? session label?) or the field will be fake.
- **H4:** Agent EOA address exposed on the dashboard (W1 short + W8 full, "same on every network") ⇒ control-plane endpoint returning the hosted-signer address; must be the Magic Server Wallet address.
- **H5:** Rebalance in-flight detail "+$6.00 arriving" ⇒ rebalancer must expose per-transfer amount + destination while pending, not just a boolean spinner.
- **H6:** Cycle countdown "in 9h 37m" + "Cycle started / Next reset (derived)" ⇒ server-computed cycle boundaries with UTC reset at 00:00; cap_reset feed banner ("The agent resumed automatically") ⇒ automatic un-halt on cycle rollover is product behavior.
- **H7:** Notification store needs: unread flag, resolved flag, sticky/awaiting-action pinning, badge = unread T2+T3 count, "Mark all as read" bulk op, meta (amount + resource) per item, and CTA typing (raise_cap / revoke / review_receipt) — matches artifact 4 plus the invented bulk-read.
- **H8:** Blocked-attempts-per-cycle counter surfaced on W4 ("2 blocked attempts this cycle (not counted)") ⇒ ledger must count blocked rows per cycle separately from spend.
- **H9:** Explorer-unavailable state ⇒ build needs explorer-link health handling (graceful degrade with copyable hash), i.e., don't hard-depend on Basescan/Arbiscan/Polygonscan availability.
- **H10:** Key lifecycle API: generate (one-time plaintext return), rotate, verify, masked display (prefix + last 4); key string format `leash_sk_<40ish base62>`.
- **H11:** npm package name `@leash/mcp` and env var `LEASH_KEY` are now user-visible contract points — the real package/env names must be decided and the snippet generated with the real key placeholder.
- **H12:** `npx leash provision` implies a provisioning CLI as a shipped artifact (or the copy must change to match the real operator flow for creating the Magic Server Wallet).
- **H13:** Overage math on W4 ("$1.20 of overage is prior settled spend — nothing was clawed back", 140%, hatched bar segment) ⇒ spend bar component must render >100% with solid-to-cap + hatched-overage segments; ledger must retain settled spend when cap is lowered.
- **H14:** Copy-to-clipboard affordances on: agent address, txHash (short + full), resource URL, trigger URL, Leash key, MCP snippet — all need real clipboard wiring + copied feedback.
- **H15:** Feed prepend animation (slide-in, newest row green tint) at ~3s polling; live pulse chips on W1/W2/W7 tied to actual connection/polling state.
- **H16:** W1 connected-agent card shows connection status derived from agent state (Active/Idle/Key rotated) — Idle while paused/frozen/halted ⇒ connection status and agent status are separate but correlated fields.

## OPEN QUESTIONS

- **Q1 (from D1):** Must the owner's unified UA treasury number be displayed (map says yes; design says no)? If yes, where — top-up section of W8 seems the natural home ("Your unified balance: $X" next to "Add funds from your unified balance").
- **Q2 (from D4/D6):** Accept the Base-first single-target top-up + auto-rebalance model, or restore the per-chain destination selector the map specifies? (Base-first is simpler and matches R-LEASH-6 async rebalancing; but the map's per-chain "Top up" was a decided-ish spec item.)
- **Q3 (from D3):** What is the HTTP wrapper tab content? Map says base URL + auth header example; spec says `@x402/fetch` wrapper (secondary interception). Needs a designed snippet (likely `import { wrapFetch } from '@leash/fetch'`-style or curl with `Authorization: Bearer leash_sk_…`). Undesigned today.
- **Q4 (from D13/H12):** Build a real `leash provision` CLI, or rewrite the W8 provisioning steps to the actual operator flow (Magic Server Wallet creation via TEE REST API)?
- **Q5 (from H3):** What is `trigger_url` truthfully capturable as in the MCP stdio proxy? If it can't be real, the field must be redefined (e.g., MCP session/tool name) or cut — a fabricated trigger URL would violate the no-diorama rule.
- **Q6 (from H1):** Verify MCP `clientInfo` actually yields "Claude Code"/"Cursor" style names (unverified); otherwise the Client row of the ACTIVE summary can't be real.
- **Q7 (from D7):** Which CONNECTION_ERROR semantics ship — 401-auth-failure (prototype, implementable) vs 24h-unused heuristic (map)? Prototype's is more honest/checkable; recommend adopting it and updating the map.
- **Q8 (OQ-W9, still open):** Does background rebalance fire a Tier-1 ambient notification? Design shows silent rebalancing (chip + row spinner only, polling-detected).
- **Q9 (from D10):** Keep "Live · ~3s polling" copy in product, or reduce to "Live"? Feed footer explainer line should almost certainly be removed (it's an annotation, not UI).
- **Q10 (from D20):** For BLOCKED rows, store/display intended settlement network (from the 402 accepts[]) instead of "—"? The cap check happens before routing, but the accepts[] is already parsed — intended network is knowable.
- **Q11:** W5 shows a resolved T3 for "Cap lowered below current spend" with no CTAs — what resolves a T3 (any owner action? cycle reset?); resolution semantics need definition for the notification store.
- **Q12:** W2 Pending rows have a txHash (0x6c8e…2b4a). Is Pending = submitted-awaiting-confirmation with hash? Matches receipt artifact status "pending"; confirm the hosted-signer flow actually has an observable pending window per chain.
