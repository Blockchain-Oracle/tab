## VERIFIED (unchanged vs our 2026-07-02 research)

### 1. v2 wire protocol headers — confirmed in published code
Grep of the extracted `@x402/core@2.17.0` tarball (`/tmp/x402check/x402-core-2.17.0/package`) finds exactly these header constants:
- `"PAYMENT-REQUIRED"` (server→client challenge, base64 JSON)
- `"PAYMENT-SIGNATURE"` (client→server payment, base64 JSON)
- `"PAYMENT-RESPONSE"` (server→client settlement receipt, base64 JSON)
- Legacy v1: `"X-PAYMENT"` and `"X-PAYMENT-RESPONSE"` still handled.

Matches `2026-07-02-x402-ecosystem-deep.md` §1. v2 remains current/default; v1 remains supported for backward compat.

### 2. `@x402/fetch` client API — confirmed from published d.ts
`/tmp/x402check/x402-fetch-2.17.0/package/dist/esm/index.d.mts`:
```ts
declare function wrapFetchWithPayment(
  fetch: typeof globalThis.fetch,
  client: x402Client | x402HTTPClient
): (input: RequestInfo | URL, init?: RequestInit) => Promise<Response>;
declare function wrapFetchWithPaymentFromConfig(fetch, config: x402ClientConfig): ...
```
Setup pattern from the doc comment in the same file:
```ts
const client = new x402Client()
  .register('eip155:8453', new ExactEvmScheme(evmSigner))
  .register('eip155:1', new ExactEvmScheme(evmSigner), 1); // trailing 1 = v1 protocol
const fetchWithPay = wrapFetchWithPayment(fetch, client);
```
This is the exact hook point for Leash's hosted-signer path: `ExactEvmScheme(evmSigner)` takes the signer, and the per-CAIP-2 `.register()` calls map directly to our Base/Arbitrum/Polygon float routing.

### 3. `@x402/mcp` — client + server wrappers confirmed in published dist

### 4. MCP SDK stdio server API for the Leash proxy — confirmed (Context7, v1.29.0 docs)
`/modelcontextprotocol/typescript-sdk/v1.29.0`:
```ts
import { McpServer } from '@modelcontextprotocol/sdk/server/mcp.js';
import { StdioServerTransport } from '@modelcontextprotocol/sdk/server/stdio.js';
const server = new McpServer({ name: 'leash-proxy', version: '...' });
await server.connect(new StdioServerTransport());
```
For the proxy (pass-through of upstream tools) the low-level `Server` + `server.setRequestHandler(ListToolsRequestSchema, ...)` / `setRequestHandler(CallToolRequestSchema, ...)` pattern is the documented approach, with an SDK `Client` connected to the upstream paid server. `@x402/mcp` requires SDK `^1.12.1`, so **1.29.0 is compatible** — pin `@modelcontextprotocol/sdk@^1.29.0`.

## CHANGED — differs from our wiki/research (the dangerous ones)

### C1. `@x402/next` requires Next.js ≥ 16.2.6 — collides with our "Next.js 14" stack decision
`npm view @x402/next@2.17.0 peerDependencies` → `{ "next": ">=16.2.6", "@x402/paywall": "^2.17.0" }`. Checked history: 2.0.0 → `next ^16.0.7`; 2.12.0 → `>=16.0.10 <16.1.0`; 2.14.0+ → `>=16.2.6`. **Every published v2 of `@x402/next` requires Next 16.** The legacy `x402-next@1.2.0` requires `next >=15.5.9 || >=16.0.10` — also not Next 14 — AND is v1-protocol, frozen since 2026-04-16, and its dist contains only `"base"` / `"base-sepolia"` network strings (no Arbitrum; grep of extracted tarball).
**Impact on Tab:** `/Users/abu/dev/hackathon/uxmax/.thoughts/decisions/DECISIONS.md` stack says "apps/web (Next.js 14)". Serving Tab's own x402 endpoint with off-the-shelf middleware on Next 14 is not possible. Options (decision needed, not made here): (a) build apps/web on Next 16.2.6+, or (b) hand-roll the 402 route handler using framework-agnostic `x402ResourceServer` from `@x402/core/server` (exported; `@x402/mcp` itself builds on it) — no Next peer dep involved. Also note the new `@x402/paywall` peer dep if we do use `@x402/next`.

### C2. Canonical repo moved: `x402-foundation/x402` — `coinbase/x402` is now a fork
GitHub API (2026-07-06): `coinbase/x402` → `"fork": true, "parent": "x402-foundation/x402", "pushed_at": "2026-06-25"`; `x402-foundation/x402` → `"pushed_at": "2026-07-06"` (active today). Published `@x402/mcp@2.17.0` package.json: `"repository": "https://github.com/x402-foundation/x402"`, `"author": "x402 Foundation"`.
**Impact:** Our clone `Reference/x402-coinbase/` is at commit dd927a2 (**2026-04-21 — 2.5 months stale**) and tracks the fork. Re-clone/re-point to `x402-foundation/x402` before mining code patterns. (Wire-format facts above were re-verified against published npm code, not the stale clone.)

### C3. Default facilitator URL correction: `https://x402.org/facilitator` (NOT `facilitator.x402.org`)
`2026-07-02-x402-ecosystem-deep.md` §4 recorded "`HTTPFacilitatorClient` pointing to `https://facilitator.x402.org` (Coinbase's hosted facilitator)". Published `@x402/core@2.17.0` contains exactly one facilitator default: `https://x402.org/facilitator`. `https://facilitator.x402.org` does not respond at all (curl: empty, no headers).
Live `GET https://x402.org/facilitator/supported` (2026-07-06) shows the open facilitator is **EVM-testnet-only**: v2 `exact`/`upto`/`batch-settlement` on `eip155:84532` (Base Sepolia) only for EVM, plus Solana/Algorand/Aptos/Stellar/Hedera entries, plus v1 `base-sepolia`. **No Base/Arbitrum/Polygon mainnet.** Mainnet settlement = CDP facilitator with API keys (see Verified §6). This also means the facilitator is no longer Coinbase-branded — it is x402 Foundation infrastructure.

### C4. MCP payment challenge now has a SECOND surface: JSON-RPC error `-32042` (SEP-1036)
Published `@x402/mcp@2.17.0` d.ts declares `MCP_PAYMENT_REQUIRED_CODE = 402` and `JSONRPC_PAYMENT_REQUIRED_CODE = -32042` ("MCP's UrlElicitationRequired JSON-RPC error code (-32042) from SEP-1036 ... ensures error.data (containing PaymentRequired) survives the McpServer round-trip, working around github.com/modelcontextprotocol/typescript-sdk/issues/774"). The dist client code extracts PaymentRequired from BOTH the tool-result path (`isError` + `structuredContent`/`content[0].text`) AND JSON-RPC errors with code 402 or -32042 (dist esm ~line 667). Our 2026-07-02 research only documented the tool-result path.
**Impact on Leash:** the MCP stdio proxy must detect payment-required on BOTH surfaces or it will miss challenges from servers built with recent `@x402/mcp` (or `McpServer`-based paid servers generally). New exported helpers exist for this: `isPaymentRequiredError`, `extractPaymentRequiredFromError`, `createPaymentRequiredError`.

### C5. Minor correction to the v1 header table in `2026-07-02-x402-detection-interception.md`
That doc's §2 table lists v1 header `X-PAYMENT-REQUIRED`. Published `@x402/core@2.17.0` contains NO `X-PAYMENT-REQUIRED` constant — v1's challenge was delivered in the JSON response body (as `2026-07-02-x402-ecosystem-deep.md` correctly states); only `X-PAYMENT` (client) and `X-PAYMENT-RESPONSE` (server) exist as v1 headers in code. cascade-proxy's `X-PAYMENT-REQUIRED` check is belt-and-braces, not spec. Leash detection: v2 = `PAYMENT-REQUIRED` header; v1 = 402 + JSON body with `x402Version: 1`.

### C6. MCP SDK v2 exists in beta under NEW package names — avoid for now
`@modelcontextprotocol/server` is at 2.0.0-beta.2 and its npm `latest` dist-tag points at the beta. This is the SDK v2 rewrite (imports like `@modelcontextprotocol/server`), distinct from `@modelcontextprotocol/sdk` (still 1.29.0, what `@x402/mcp` depends on). Danger: docs/snippets from "main" branch now show v2 package paths. Pin `@modelcontextprotocol/sdk@^1.29.0` and ignore v2 packages for Leash v0.

## Direct answers to the five questions

1. **`@x402/core` / `@x402/fetch` / `@x402/evm` / `@x402/mcp`** — all `2.17.0` (2026-06-26). Protocol v2 (`PAYMENT-REQUIRED`/`PAYMENT-SIGNATURE`/`PAYMENT-RESPONSE` headers) is current/default; v1 (`X-PAYMENT`, body-based challenge) still supported via `.register(network, scheme, 1)`.
2. **x402 on Next.js for Tab** — use `@x402/next@2.17.0`, BUT it requires Next ≥16.2.6 (+ `@x402/paywall` peer). Arbitrum support comes from the protocol/facilitator layer, not the middleware: CAIP-2 `eip155:42161` works when settled via the CDP facilitator. Legacy `x402-next` is v1, frozen, Base-only, and still not Next-14-compatible. See C1 for options.
3. **`@modelcontextprotocol/sdk`** — `1.29.0`. Stdio server: `McpServer` (or low-level `Server`) + `StdioServerTransport` from `@modelcontextprotocol/sdk/server/{mcp,stdio}.js`; proxy pattern = low-level `Server` with `setRequestHandler(ListToolsRequestSchema|CallToolRequestSchema)` forwarding to an SDK `Client` connected upstream. Compatible with `@x402/mcp` (needs `^1.12.1`).
4. **CDP facilitator** — `https://api.cdp.coinbase.com/platform/v2/x402`; Base, Arbitrum One, Polygon (and World) mainnets ACTIVE with `exact`/`upto`/`batch-settlement`; CDP API keys required; 1,000 free tx/mo then $0.001/tx. The open `https://x402.org/facilitator` is testnet-only for EVM.
5. **Native USDC (EIP-3009)** — Base `0x833589fCD6eDb6E08f4c7C32D4f71b54bdA02913`; Arbitrum One `0xaf88d065e77c8cC2239327C5EDb3A432268e5831`; **Polygon `0x3c499c542cEF5E3811e1192ce70d8cC03d5c3359`**. All EIP-712 domain "USD Coin"/"2", 6 decimals, EIP-3009 default in the SDK.
