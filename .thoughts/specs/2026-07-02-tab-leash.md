# Spec: Tab + Leash

## Objective

Tab + Leash is one submission on one payment rail, targeting three prizes: the Particle Network
flagship Universal Accounts track ($2,500), the Arbitrum settle bonus ($2,000), and the Magic
embedded wallet bonus ($500). **Tab** is a Stripe-style embeddable crypto pay-button SDK — a
merchant drops in a `<PayButton>`, creates a server payment-intent endpoint that pins
`receiver`, `token`, and `chainId: 42161` (Arbitrum One), and receives settlement via webhook.
A buyer logs in with Magic email OTP; their EOA becomes a Particle Universal Account in
EIP-7702 mode (`smartAccountOptions.useEIP7702: true`), pays from their unified cross-chain
balance, and sees no chain/gas/bridge words. **Leash** is an x402 auto-payer for the owner's
own external AI agent (no LLM, no model picker — the owner brings their agent). When the agent
hits an HTTP 402 paywall, Leash intercepts the call via an MCP stdio proxy (primary) or an
`@x402/fetch` wrapper, pays from a pre-positioned USDC float held in a hosted-signer wallet
(Magic Server Wallet TEE) within a human-set cap, and the agent continues — with no code
changes required in the agent itself. Floats are pre-positioned on Base (primary, ~75% of real
x402 traffic) and Arbitrum (~6%); Polygon is excluded from v0 — UA SDK V2 removed Polygon
rebalance support (see Constraint 14). A Particle UA treasury rebalances floats
asynchronously in the background so the hot path is never blocked by a bridge. The unity: Tab
is the receiving side of the rail; Leash is the paying side. Tab can optionally expose its
checkout as an x402 resource that Leash pays headlessly — one rail, two payer types (human via
modal, agent via x402).

## Background And Current Reality

### Payment rail mechanics (verified)

x402 settles atomically: the CDP hosted facilitator calls EIP-3009
`transferWithAuthorization` on the exact network named in the 402 and does an on-chain
`balanceOf(from)` before settling, so USDC must already sit at the payer's address on that
chain. The Particle Universal Account's chain-abstracted unified balance is invisible to the
facilitator. EIP-3009 uses `ecrecover` — only a key-holding EOA address qualifies; standard
USDC v2 rejects EIP-1271 smart-contract signatures (ERC-7598 is proposed, not shipped). A UA
in Smart Account (ERC-4337) mode CANNOT pay x402. A UA in EIP-7702 mode CAN pay x402 because
in 7702 mode the UA address IS the owner EOA (same address, key intact, no new contract
deployed), producing a valid EIP-3009 signature. Source:
`research/x402-tab-leash-mechanics.md §2` (all items marked ✅ verified).

The realistic payer pattern: the agent holds a small USDC float on Arbitrum One (`eip155:42161`) at its EOA address
and signs EIP-3009 directly per x402 call. The Particle UA SDK is used separately — and only
— to top up / replenish that float; the two flows do not compose into a single payment call.
(Verify during the Story-9 spike: USDC on Arbitrum One supports EIP-3009 transferWithAuthorization AND the CDP facilitator settles x402 on Arbitrum end-to-end.)
Source: `research/x402-tab-leash-mechanics.md §2` (Option A + Option B).

### Particle UA-7702 integration (verified)

`createTransferTransaction({ token: { chainId: CHAIN_ID.ARBITRUM_MAINNET_ONE, address: TOKEN }, amount, receiver: MERCHANT })` accepts an arbitrary third-party receiver and returns `{ rootHash, userOps }`. Signing requires two conceptual steps: the 7702 authorization(s) extracted from `userOps` (via `userOp.eip7702Auth`, de-duped per execution by `(chainId, nonce, authorization address)`) plus the `rootHash` (`personal_sign`). `sendTransaction(transaction, signature, authorizations)` must receive the `authorizations` array as the third argument in 7702 mode — omitting it on an undelegated chain fails. The nested `smartAccountOptions.useEIP7702: true` init form (not the flat quickstart form) is required to prove 7702 mode to judges. Source: `wiki/sdks/particle-universal-accounts.md` (init section, gotchas, and full 7702 browser flow).

Supported 7702 providers are a verified short list: Dynamic (WaaS), Magic, and Privy — plus raw
server keys. MetaMask / extension / JSON-RPC wallets are explicitly unsupported in 7702 mode.
Source: `wiki/sdks/particle-universal-accounts.md` (surface support section).

External-receiver settlement (`createTransferTransaction` to a merchant, not to the user's own
UA) is **doc-proven** (transfer.mdx, web-quickstart.mdx, Context7 API) but **NOT yet run
live** — the official cloned Privy demo only exercises `createBuyTransaction` /
`createSellTransaction` into the user's own UA. A funded live spike is required before the
demo. Source: `research/x402-tab-leash-mechanics.md §4(a)` (marked ⚠️ unverified live).

UA V2 migration is in progress; users may need to withdraw funds from old accounts.
`UNIVERSAL_ACCOUNT_VERSION` must be verified at build time. Source:
`wiki/sdks/particle-universal-accounts.md` (gotchas).

### Magic embedded wallet (verified)

Magic `loginWithEmailOTP({ showUI: false, deviceCheckUI: false })` (headless) dispatches an
OTP email; Tab renders the 6-box screen and passes the code to `verifyEmailOTP`. On success,
a DID token is issued; `magic.user.getInfo()` yields the plain EOA address. That EOA is
passed as `ownerAddress` (inside `smartAccountOptions`) to the Particle UA init. Magic's embedded client SDK is a valid 7702-mode
signer (referenced in `ua-7702-magic-demo`). The Magic $500 bonus requirement is literally
"embedded wallet"; the buyer-facing Magic client SDK satisfies this. Server Wallets (TEE REST
API) are a headless alternative for the agent-key path but are NOT the same product as the
embedded wallet. Source: `wiki/sdks/magic.md` (embedded wallet section, bounty scope,
gotchas).

### UX current reality (verified patterns from shipped products)

- **One-tap is the dominant win pattern.** Apple Pay (Face ID), Daimo (single source
  transfer), Stripe Link (email recognized → OTP → tap) collapse the flow to one
  authenticate/confirm. Source: `research/2026-06-30-ux-reality-research.md §A, inference 1`.
- **Optimistic instant-success beats waiting.** Daimo shows success on `payment_started`
  (source tx detected) before `payment_completed`; OnchainKit label morphs to "View payment
  details" on state transition — not after block confirmations. Coinbase Commerce hosted is
  the named anti-pattern (countdown + "waiting for confirmations"). Source: `research/2026-06-30-ux-reality-research.md §A, inference 2`.
- **The pay button IS the state machine.** Idle (CTA + amount) → processing (disabled +
  spinner) → success (label morphs / animated checkmark). Recurs across Stripe, OnchainKit,
  Apple Pay, Shop Pay. Source: `research/2026-06-30-ux-reality-research.md §A (button-state
  sequence table), inference 3`.
- **Silent wallet creation; celebrated funding completion.** No wallet-creation screen across
  Privy, Dynamic, Magic, Coinbase, Phantom. Funding completion gets an explicit confirmation
  ("Purchase complete!"). Source: `research/2026-06-30-ux-reality-research.md §B (wallet
  provision pattern), inference 4`.
- **Hide all chain/gas/bridge/fee language.** Explicit in Daimo; consistent across all
  embedded wallet auth screens. Source: `research/2026-06-30-ux-reality-research.md §B
  (vocabulary pattern), inference 5`.
- **Six-box OTP with auto-submit is de facto standard.** Privy and Dynamic converged on
  identical structure independently: back arrow → envelope icon → 6 boxes → auto-advance →
  auto-submit on 6th digit → resend link. Source: `research/2026-06-30-ux-reality-research.md §B (OTP screen)`.
- **No incumbent agent-spend dashboard.** No shipped first-party consumer "agent spend
  screen" was found. The pattern is assembled from: per-payment receipt data + read-only
  balance + app-layer control (pause/revoke). Payman, Skyfire, and Coinbase CDP Portal are
  establishing this space in real time. Source: `research/2026-06-30-ux-reality-research.md §C, inference 9`.
- **Four-level revocation spectrum.** Soft pause (resumable, credential intact) → freeze
  (temporarily inactive, resumable) → cancel/archive (permanent, new provisioning needed) →
  nuclear (network/delegation-level, no recovery without re-provisioning). Source:
  `research/2026-06-30-ux-reality-research.md §C (revocation spectrum)`.
- **Three-tier notification model.** Silent (routine, log only) / banner (approaching limit,
  unusual event) / interrupt (over budget, approval required). Rule: never use the interrupt
  tier for routine events. Source: `research/2026-06-30-ux-reality-research.md §C
  (notification tiering)`.

## Users

**Merchant / Developer:** integrates Tab SDK into an existing web app to accept crypto
payments without crypto expertise. Wants a drop-in button, a server endpoint, and a webhook —
the Stripe mental model. Has a receiving address and a preferred token/chain. Does not want to
run a bridge, manage a liquidity pool, or educate buyers about gas.

**Buyer:** wants to complete a purchase with their email address and nothing else. Has no
pre-existing Arbitrum balance, no installed wallet extension, no knowledge of which chain they
are on. May hold ETH on Base, USDC on Polygon, or any supported primary asset on any
supported chain. Expects the payment to feel like a card transaction.

**Agent Owner / Operator:** has deployed or is running their own external AI agent (Claude
Code, Claude Desktop, Cursor, or any HTTP-capable agent) that needs to pay for internet
resources (APIs, data, compute) autonomously and at scale. Connects the agent to Leash to add
x402 auto-pay without touching the agent's code. Wants to set a spending ceiling, see every
payment in real time, and revoke access instantly from a phone if something goes wrong. Does
not want the agent to hold unlimited authority, and does not want to re-code the agent to add
payment capability.

## Goals

1. **Prove the flagship UA-7702 requirement.** The human buyer's Magic-email EOA becomes a
   Particle Universal Account in `useEIP7702: true` mode and executes at least one
   cross-chain value move that settles on Arbitrum One (`chainId: 42161`).

2. **Make crypto invisible for the buyer.** From button click to success screen, no word
   "chain", "gas", "bridge", "Arbitrum", "EOA", or "wallet address" appears in the
   buyer-facing UI. The buyer's only inputs are an email address and a 6-digit OTP code.

3. **Demonstrate the agent payment rail.** Leash intercepts x402 paywall responses for the
   owner's external agent and auto-pays from pre-positioned USDC floats on Base (primary),
   Arbitrum, and Polygon, routing per the resource's `accepts[].network` field, within an
   owner-set cap, with no human intervention per payment and no code change in the agent.

4. **Give the agent owner real-time visibility and layered control.** A web spend dashboard
   shows a live payment feed, a spend bar (current / cap), three-tier notifications, and the
   full revocation spectrum (soft pause / freeze / cancel / nuclear). A mobile PWA mirrors
   this with push notifications and one-tap revoke.

5. **Connect both payer types through one rail.** Tab's merchant-checkout SDK accepts both
   human payers (via the Magic + UA-7702 modal) and agent payers (via x402 headless payment)
   settling the same token on Arbitrum from the merchant's perspective.

6. **Prove the flagship UA cross-chain move via background float rebalancing.** The Particle
   UA treasury rebalances pre-positioned floats with real `createTransferTransaction` calls
   (e.g., funds on Ethereum / Polygon → Base USDC at the agent address), logged to the
   universalx.app explorer. This is the flagship track's required cross-chain value move on the
   Leash side; the buyer's Tab checkout delivers the same qualification on the human-payer side.

7. **Stack the three prizes cleanly and non-cosmetically.** Flagship UA track: buyer's
   Magic-email EOA executes a real cross-chain 7702 op AND the Leash UA treasury rebalances
   cross-chain. Arbitrum settle: merchant token lands on Arbitrum One. Magic embedded wallet:
   buyer authenticates via the Magic client SDK (`magic-sdk`, `loginWithEmailOTP`); Leash agent
   wallet uses Magic Server Wallets (TEE hosted signer).

## Non-Goals

- **Extension / injected-wallet UA-7702.** MetaMask and JSON-RPC wallets cannot produce
  `signAuthorization` / Type-4 authorization lists; 7702 mode is explicitly unsupported.
  Source: `wiki/sdks/particle-universal-accounts.md` (surface support warning).

- **LLM / model selection.** Leash has no LLM, no inference call, and no model picker. The
  owner connects their own external agent (Claude Code, Cursor, or any HTTP agent); Leash is
  the payment layer only.

- **On-demand cross-chain top-up in the x402 hot path.** Particle UA `createTransferTransaction`
  takes 10–90 seconds end-to-end (cross-chain bridge latency). x402 `maxTimeoutSeconds` can be
  as low as 60 seconds and the EIP-3009 `validBefore` deadline cannot be paused during a
  bridge. On-demand bridging would block every x402 call and would fail on tight-timeout
  resources. Pre-positioned floats with async background rebalancing is the only viable
  architecture. Source: `research/2026-07-02-x402-multichain-strategy.md §2`.

- **Solana x402 support (v0 out of scope).** EIP-7702 is EVM-only. x402 on Solana uses the
  `exact_svm` scheme, `@x402/svm`, and Ed25519 keypairs — a separate code path with a separate
  signer. Solana hard-limits `maxTimeoutSeconds` at ~80–90 s (blockhash expiry). Solana does
  not add to the Particle UA Track or Arbitrum prize stack. Excluded from v0; may be a future
  parallel SVM path. Source: `research/2026-07-02-x402-multichain-strategy.md §5`.

- **On-chain session-key spend cap.** Particle session keys are Biconomy-v2.0.0-only and
  cannot attach to a UA in 7702 mode. The Leash spend cap is enforced server-side in the
  hosted signer (Magic Server Wallet TEE): the signer rejects signing requests once cumulative
  spend reaches the cap (returns 403). No `SpendCapGuard.sol` or on-chain delegation contract
  is in scope. Source: DECISIONS.md; `research/x402-tab-leash-mechanics.md §4`.

- **Hosting an x402 facilitator.** The agent is always the payer; the seller's facilitator
  (CDP hosted: `https://api.cdp.coinbase.com/platform/v2/x402`) handles settlement. Source:
  `research/x402-tab-leash-mechanics.md §1`.

- **Native iOS / Android mobile signing.** No documented UA-7702 path exists for native
  mobile or React Native. The mobile surface signs nothing and holds no private key. Source:
  `wiki/sdks/particle-universal-accounts.md` (mobile = partial/unverified); `research/x402-tab-leash-mechanics.md §5`.

- **Throwaway multi-surface demo.** Browser-extension 7702 is dead on the flagship track.
  Adding surfaces for surface count is explicitly out of scope; each surface must strengthen
  the demo or be cut.

- **Chain / gas / token selection UI for the buyer.** The product's core proposition is
  hiding these entirely. A buyer who must pick a network is the Coinbase Commerce anti-pattern.

- **Overclaiming cryptographic delegation.** The Leash "leash" is economic (capped funding)
  and operational (revoke/freeze), not a Particle session key or an on-chain cryptographic
  primitive. The pitch must be honest; judges penalize overclaiming. Source:
  `research/ideas/leash.md` (brutal honesty section).

- **CDP / Coinbase AgentKit dependency.** CDP is only relevant if Tab hosts its own x402
  endpoint or uses CDP server wallets — neither is committed. Removed from required keys.
  Source: DECISIONS.md (CDP — CORRECTED).

## Requirements

### Tab SDK / Checkout

**R-TAB-1 — `<PayButton>` drop-in component.**
The merchant installs the `packages/sdk` package and drops in a single `<PayButton>`
component. The component is self-contained: it renders the idle state, manages auth, drives
the payment state machine, and shows success or error — all without the merchant writing
payment logic. The component requires `apiBaseUrl`, the Tab `publishableKey` (`pk_*`, not the
Magic publishable key), `intentUrl` (the merchant relay endpoint), and
`onSuccess(transactionId, tokenChanges)` as props. Source: `research/ideas/tab.md`
(merchant integrates X); UX research §A (SDK DX converges on "single component + server
intent + webhook").

**R-TAB-2 — Server payment-intent endpoint.**
The merchant server calls sk-auth `POST /api/v1/payment-intents` with exact
`{amount,intentUrl}`. Tab derives the merchant, environment, current receiving address, and
fixed settlement asset, then returns `{intent,intentToken}` with a five-minute signed JWS.
The merchant's `intentUrl` relays that response to checkout; the Phase 5 demo relay is not
built in Phase 4. Checkout refetches on click and opens through pk-auth
`POST /api/v1/payments` with exact `{intentToken}` only. Browser-supplied payment fields are
never authoritative. Source: `research/x402-tab-leash-mechanics.md §3` (Stripe analog:
server-side amount + pinned receiver + asset) and the 2026-07-16 authority decision.

**R-TAB-3 — Webhook delivery (canonical contract).**
The Tab server POSTs to the merchant's configured webhook URL only after inline chain
verification completes (see R-TAB-11). The merged webhook payload is:
```json
{ "id": "<delivery_id>", "type": "payment.settled", "livemode": true,
  "transactionId": "...", "tokenChanges": [...] }
```
The request carries `X-Tab-Signature: t=<unix_timestamp>,v1=<HMAC-SHA256(whsec, "<t>.<rawBody>")>`.
Delivery policy: exactly three total HTTP attempts. Attempt 1 dispatches inline immediately
after verified settlement; failures schedule attempt 2 after 1 minute and attempt 3 after
4 minutes. Each attempt has a 10-second response timeout. A third failure immediately records
terminal `gave_up`; there is no 16-minute delay and no fourth request. The `webhook_deliveries`
table is the single source of truth for delivery badges (M4 transactions list) and the delivery
log (M6). The "≤15 s
first-attempt" latency target is an internal goal contingent on hosting decisions and must not
appear in merchant-facing documentation until confirmed. Source: reintegration B-4, B-5.

**R-TAB-4 — Magic email-OTP buyer authentication (headless).**
The checkout modal authenticates the buyer via
`magic.auth.loginWithEmailOTP({ showUI: false, deviceCheckUI: false })` (headless form).
Tab renders the 6-box OTP screen (auto-advancing on each digit, auto-submitting on the 6th);
Magic dispatches the OTP email and validates the code server-side. The buyer enters only an
email address and a 6-digit code. No MetaMask, no seed phrase, no external URL is visited.
Checkout first tries the persistent path `isLoggedIn()` → `getIdToken()` + `getInfo()` and
uses `getInfo().wallets.ethereum.publicAddress`; OTP is fallback only. Both paths are
server-bound to the submitted email through Magic Admin DID validation. Capability verified
against Magic 33.9.0 installed types and the Phase 3 real session flow. Source: reintegration A-5;
`wiki/sdks/magic.md`; UX research §B (6-box auto-submit as de facto standard).

**R-TAB-5 — Particle UA-7702 initialization.**
The buyer's Magic EOA address is passed as `ownerAddress` into:
```ts
new UniversalAccount({
  projectId, projectClientKey, projectAppUuid,
  smartAccountOptions: {
    useEIP7702: true,
    name: "UNIVERSAL",
    version: UNIVERSAL_ACCOUNT_VERSION,
    ownerAddress: publicAddress,   // inside smartAccountOptions only — no top-level field
  },
  tradeConfig: { slippageBps: 100 }, // V2-valid; universalGas is removed
})
```
The nested `smartAccountOptions.useEIP7702: true` form is mandatory (not the flat quickstart
form) to prove 7702 mode to judges. Source: `wiki/sdks/particle-universal-accounts.md`
(init, gotcha about two init shapes).

**R-TAB-6 — Unified balance display in checkout modal.**
After auth, the modal reads `ua.getPrimaryAssets()` and displays `totalAmountInUSD` as a
single USD balance. No per-chain breakdown is shown. The buyer sees one number. Source:
`wiki/sdks/particle-universal-accounts.md` (core concepts, unified balance); UX research §B
(hide chain/gas language); UX research §A (total price visible before auth — Google Pay rule).

**R-TAB-7 — Cross-chain transfer settlement on Arbitrum.**
Payment is executed via:
```ts
const tx = await ua.createTransferTransaction({
  token: { chainId: CHAIN_ID.ARBITRUM_MAINNET_ONE, address: TOKEN },
  amount,
  receiver: MERCHANT,
});
// sign authorizations from tx.userOps + rootHash
await ua.sendTransaction(tx, rootHashSignature, authorizations);
```
The `authorizations` array is built from `tx.userOps` (each `userOp.eip7702Auth` not yet
delegated), de-duped per execution by `(chainId, nonce, authorization address)`. The UA needs no gas or token pre-held on Arbitrum. Source:
`wiki/sdks/particle-universal-accounts.md` (createTransferTransaction snippet, full 7702
flow, gotcha about 3rd arg); `research/x402-tab-leash-mechanics.md §3`.

**R-TAB-8 — Optimistic success state.**
The checkout modal transitions to an animated success state immediately when `ua.sendTransaction`
returns a `transactionId` — not after polling for on-chain block confirmations. The buyer
does not wait for finality. Source: UX research §A (crypto-specific 3-state model: state 2
"fires immediately on tx detection, NOT after confirmations"; Daimo `payment_started`
optimistic pattern; inference 2 "optimistic instant-success beats truthful blockchain-exposed
waiting").

**R-TAB-9 — Zero crypto vocabulary in buyer-facing UI.**
None of the following words appear in any buyer-facing string, copy, or tooltip: "chain",
"gas", "bridge", "Arbitrum", "EOA", "wallet address", "token", "sign", "EIP", "USDC" (unless
shown as a currency unit the merchant opted into). Source: UX research §B (vocabulary
pattern: "Gas and chain are absent from onboarding"); UX research §A, inference 5 ("Hide all
chain/gas/bridge/fee language").

**R-TAB-10 — Button state machine.**
The `<PayButton>` progresses through: IDLE ("Pay $X.XX" + lock icon) → LOADING (disabled +
spinner) → AUTH (Magic OTP modal, separate from button) → PROCESSING (disabled, "Processing…"
or spinner, prevents double-submit) → SUCCESS (animated checkmark, morphs to "Done" or
merchant-configured label) → ERROR (inline message, entered data preserved, retry in place).
Source: UX research §A (button-state sequence table, OnchainKit lifecycle, Stripe form shake
on failure).

**R-TAB-11 — Canonical payment lifecycle.**
The Tab backend follows a create-at-open lifecycle with three phases:

1. **Open** — `POST /api/v1/payments` (pk-auth) accepts exact `{intentToken}` only and rejects
   raw browser payment fields. It verifies the JWS signature, merchant, environment, replay
   ID, expiry, and current receiving address, then mints a server-side `ref_code` and creates
   a `payments` row (`status: pending`). The payment exists before any on-chain operation.

2. **Report** — `PATCH /api/v1/payments/:id` (pk-auth, buyer-originated): carries
   `{ transactionId, tokenChanges, buyerDidToken }`. The server validates `buyerDidToken`
   via Magic Admin SDK and stores the verified `payer_address`. Phase 3 intentionally leaves
   `payer_email` NULL and omits it from API/dashboard responses while D5 remains open. It runs inline chain verification (Particle tx
   lookup / `TransactionResult` dest-hash, disambiguated by the buyer's known UA sender
   address — never amount + time-window alone); on confirmation transitions to `status:
   settled` and fires the webhook (R-TAB-3). A cron backstop sweep (`GET
   /api/internal/settle-sweep`, ~1/min) re-verifies rows stuck in `pending` beyond a
   threshold, covering network failures that blocked the PATCH call.

Tables: `payments` + `settlements`. Receipt status enum: `pending | settled | failed |
blocked`. The `ref_code` minted at open is the same reference shown on the buyer's success
screen (Reference #TAB-…) and in the merchant's M4 transactions list. Source: reintegration
B-2, B-3, A-14.

**BLOCKED (B-04):** the buyer live-settle path (`sendTransaction` → PATCH → webhook) is
blocked on the funded live spike. The lifecycle scaffolding (routes, tables, ref_code mint)
is plannable now. No code path may mark a receipt `settled` without a real on-chain confirmation.

**R-TAB-12 — Add-funds surface (buyer money-in).**
When the checkout modal reaches the `INSUFFICIENT_BALANCE` or `INSUFFICIENT_BALANCE_ERROR`
state (`totalAmountInUSD < payment amount`), the modal surfaces a funding path instead of a
dead-end:

- Displays the buyer's real UA deposit address (live read post-auth from the provisioned UA).
- Provides a copy-to-clipboard affordance.
- Shows honest copy: "Send USDC on a supported network to this address."
- Optionally links to universalx.app for a guided on-ramp.

Cancel → returns to IDLE; re-check fires on the next click. Until this surface ships,
**the demo buyer must be pre-funded by the operator and the demo script + deck must state
this explicitly** (deviation D23). Source: reintegration A-12, skeptic BLOCKER #1.

**R-TAB-13 — Secret-key API plane.**
A server-side API plane authenticated by `sk_` keys provides programmatic payment access:

- `GET /api/v1/payments` — list payments for the authenticated merchant; supports
  `env=test|live` filter.
- `GET /api/v1/payments/:id` — retrieve a single payment record.

sk-auth middleware enforces permissions and stamps `last_used_at` on each request. Permission
labels map to enforced capabilities: `Full access` (read + manage) / `Read-only — list and
read payments`. This is the plane `new Tab(sk)` exercises in the Quickstart snippet, making
the API keys table's "LAST USED" column real rather than always "Never". Source: reintegration
B-6, skeptic MAJOR SD-I.

### Leash Agent

**R-LEASH-1 — Hosted signer + pre-positioned multi-chain USDC floats.**
The Leash agent wallet is a **hosted signer**: a Magic Server Wallet (TEE REST API,
`@magic-sdk/admin` / `magic-sdk` server SDK). The signer holds the agent's private key in a
TEE — the key is never exposed to `apps/agent` code, client, or env var. The hosted signer
enforces the owner-set spend cap **server-side**: when cumulative settled spend would exceed
the cap, the signer rejects the signing request (returns a 403-equivalent error) before any
EIP-3009 authorization is issued. The corresponding EOA holds pre-positioned USDC floats on:

| Chain | CAIP-2 | Float target | Rationale |
|---|---|---|---|
| Base | `eip155:8453` | $20–50 | ~75% of real x402 resources; protocol default |
| Arbitrum One | `eip155:42161` | $10–20 | Tab merchant endpoints; ~6% of ecosystem; ARB bounty chain |

Polygon (`eip155:137`) is **dropped from v0**: UA SDK V2 removed Polygon rebalance support,
leaving Polygon as a money-in / no-money-out dead-end. See Constraint 14 and reintegration I-8.

**Money-movers BLOCKED (B-04):** top-up, auto-rebalance, withdraw, and agent mainnet pay all
require the funded live spike (Task #2). No code for these states may mark itself done until
the spike report exists.

Floats are pre-positioned before the agent starts. On-demand bridging in the hot path is
explicitly forbidden (too slow — see Constraints). Source:
`research/2026-07-02-x402-multichain-strategy.md §3` (Architecture B, recommended);
`research/2026-07-02-x402-payer-architecture.md §Pattern 1`; `wiki/sdks/magic.md` (Server
Wallets TEE section).

**R-LEASH-2 — Interception: MCP stdio proxy (primary) + HTTP fetch-wrapper (secondary) + MITM (last resort).**

**Primary (~90% of target traffic) — MCP stdio proxy:**
`apps/agent` ships an MCP stdio server built with `@modelcontextprotocol/sdk` and
`@x402/mcp`. It runs as an MCP server entry in `claude_desktop_config.json` /
`cursor_mcp_config.json` / Claude Code settings. When the owner's agent (Claude
Code/Desktop, Cursor, OpenClaude) makes a tool call that hits an x402-gated MCP resource,
the proxy intercepts at the MCP protocol layer, pays via the hosted signer, and retries
transparently. No CA cert required. This is the zero-code-change path for the majority of
real users.
```jsonc
// claude_desktop_config.json (example install)
{
  "mcpServers": {
    "leash": { "command": "npx", "args": ["leash-mcp"], "env": { "LEASH_API_KEY": "..." } }
  }
}
```

**Secondary — HTTP fetch-wrapper (`@x402/fetch`):**
For agents that make direct HTTP calls (not via MCP), `apps/agent` can export an
`@x402/fetch`-powered wrapper. The owner imports `wrapFetchWithPayment` once at startup;
every subsequent HTTP call that receives a 402 is handled transparently. Wallet calls route
to the hosted signer.
```ts
import { wrapFetchWithPayment } from "@x402/fetch";
const fetchWithPay = wrapFetchWithPayment(fetch, leashPaymentClient);
```

**Last resort — Transparent MITM proxy + CA cert (opt-in):**
For black-box runtimes (Docker containers, shell scripts, non-JS agents) that cannot use MCP
or the fetch wrapper, `apps/agent` provides an optional local HTTP/HTTPS forward proxy. The
agent sets `HTTP_PROXY=http://localhost:9402`. The proxy holds the hosted signer credentials.
Requires a one-time CA cert installation. Not in v0 scope unless another interception method
fails to cover the demo; kept as a fallback.

In all three modes the agent never calls `/verify` or `/settle` directly — the seller's
facilitator (CDP hosted) handles on-chain settlement. Source:
`research/2026-07-02-x402-payer-architecture.md §Pattern 2` (MCP primary);
`research/2026-07-02-x402-interception-feasibility.md §Mode B` (MCP stdio proxy internals);
`research/2026-07-02-x402-payer-architecture.md §Pattern 1` (fetch-wrapper secondary);
`research/2026-07-02-x402-payer-architecture.md §Pattern 3` (proxy last resort).

**R-LEASH-3 — Hosted-signer spend cap (server-side, outside the agent).**
The cap is enforced by the Magic Server Wallet (hosted signer), not inside the agent loop.
Before issuing an EIP-3009 signature for any x402 payment, the signer checks cumulative
settled spend from the persistent receipt ledger. If the proposed payment would push
cumulative spend over the owner-set cap, the signer rejects the signing request and returns
an error; the Leash middleware surfaces `LEASH_CAP_EXCEEDED` to the interceptor, logs the
blocked attempt, and emits a Tier 3 notification. Because the cap gate lives **outside the
agent**, it cannot be bypassed by the agent process itself — overspend produces a 403-class
rejection at the signer, not an agent-side policy check. The cap is NOT an on-chain smart
contract. Source: corrected architecture; `research/x402-tab-leash-mechanics.md §4` (leash =
app-layer policy); DECISIONS.md (no custom contract).

**R-LEASH-4 — Per-payment receipt logging.**
After every x402 settlement attempt, the Leash middleware writes a receipt row to the
persistent store with: amount (atomic units + USD-denominated display), asset, network
(CAIP-2, e.g. `eip155:8453`), `payTo` (resource endpoint), `txHash` (from
`X-PAYMENT-RESPONSE`), timestamp, trigger URL, and `success: true | false`. Failed or
blocked payments are logged with `success: false` and a `reason` field (`CAP_EXCEEDED`,
`FLOAT_EMPTY`, `SIGNER_REJECTED`, etc.). The receipt ledger is the source of truth for the
hosted signer's cap check (R-LEASH-3). Source: `research/x402-tab-leash-mechanics.md §1`
(`X-PAYMENT-RESPONSE` receipt fields: `success`, `txHash`, `network`); UX research §C
(settled receipt row state).

**R-LEASH-5 — Per-payment chain routing by CAIP-2 network.**
Before signing any x402 payment, the Leash middleware parses the 402 response's
`accepts[].network` field (CAIP-2, e.g. `eip155:8453` for Base, `eip155:42161` for
Arbitrum). It selects the matching pre-positioned float address and routes the EIP-3009
signing request to the hosted signer for that chain. If no matching float exists or the float
is empty, the middleware surfaces `LEASH_FLOAT_EMPTY` to the interceptor (does not block the
overall agent), logs the miss, and triggers a background float top-up (R-LEASH-6). The agent
never sees chain selection; routing is entirely internal to Leash. Source:
`research/2026-07-02-x402-multichain-strategy.md §3` (CAIP-2 routing steps 1–6).

**R-LEASH-6 — Async UA treasury → float rebalancing (background, never hot-path).**
`apps/agent` runs a background watcher that polls each pre-positioned float balance. When any
float drops below 25% of its target, it issues a non-blocking `createTransferTransaction` via
the Particle UA SDK to top up that chain from the unified UA treasury:
```ts
const topUp = await ua.createTransferTransaction({
  token: { chainId: CHAIN_ID.BASE, address: BASE_USDC_ADDRESS },
  amount: "20",
  receiver: agentAddress,  // self-top-up; agentAddress == UA address in 7702 mode
});
const sig = await hostedSigner.signMessage(getBytes(topUp.rootHash));
await ua.sendTransaction(topUp, sig);
// result.transactionId links to universalx.app/activity/details?id=...
```
This call is **never on the critical path** of an x402 payment. If the UA top-up is in
flight, the current float is used for x402 calls until replenished. The top-up latency (10–90
seconds cross-chain) does not affect agent-facing payment latency. Each successful top-up is
a real Particle UA cross-chain value move that satisfies the flagship track requirement.
Source: `research/2026-07-02-x402-multichain-strategy.md §3, §4` (float thresholds, UA
`createTransferTransaction` code snippet); `wiki/sdks/particle-universal-accounts.md`
(createTransferTransaction call shape).

### Spend Dashboard + Mobile Monitor

**R-DASH-1 — Live payment feed.**
The web dashboard (`apps/web`) shows a real-time feed of x402 payment receipts in
reverse-chronological order. Each row shows: amount (USD-denominated), resource / URL,
timestamp, `txHash` (linked to block explorer), settlement status (`success` / `failed`).
Source: UX research §C (settled receipt row; Payman Execution Trace — per-entry fields:
timestamp + counterparty + status badge).

**R-DASH-2 — Spend bar + cap configuration.**
The dashboard shows a spend bar (current cumulative spend / cap, percentage). The owner can
set the cap (amount + reset frequency). The cap enforces a hard block when hit — payments
halt, not just notify. Threshold notifications fire at 75% (Tier 2 banner) and 100% (Tier 3
interrupt). The cap must be set BELOW the operator's true maximum tolerated spend to absorb
any notification lag (which can reach "several minutes" per verified Vercel + Cloudflare
documentation). Source: UX research §C (Vercel 50%/75%/100% thresholds; Cloudflare hard
block at limit; latency caveat).

**R-DASH-3 — Three-tier notifications.**
- **Tier 1 (silent / ambient):** every settled payment logged; badge count updates. No alert
  pushed to the owner.
- **Tier 2 (banner):** approaching cap (75%), unusual or unrecognized resource URL.
- **Tier 3 (interrupt + require action):** cap exceeded, agent halted, suspicious activity.

Tier 3 must never be used for routine events. Source: UX research §C (notification tiering,
anti-cry-wolf rule; Smashing Magazine / Mantlr convergence).

**R-DASH-4 — Revocation spectrum (canonical four-level model).**
Four discrete revoke actions are accessible from both the web dashboard and mobile PWA via
`POST /api/leash/revoke`. No on-chain transaction is used at any level.

1. **L1 — Pause / Resume (resumable):** status flip → `paused`. The agent-plane pre-parse
   halts all outbound payment signing; the signer credential is untouched. Owner can resume.
   Copy uses payment-stop phrasing (Leash does not run the agent's own process loop).

2. **L2 — Freeze / Unfreeze (resumable):** status flip → `frozen`. The signer gate blocks
   all signing requests in addition to the L1 halt; the credential is untouched and
   resumable.

3. **L3 — Cancel:** Revokes the `signer_subject` at the Tab OIDC credential layer and
   invalidates all Leash keys. **Magic has no wallet-delete or key-rotate endpoint** (verified
   absent at Reference/magic-docs `45d1eb2`); cancellation is entirely at our credential layer.
   Re-provisioning mints a new OIDC subject. **Named spike item (B-03, unverified):** whether
   a new OIDC subject yields a new TEE wallet address for the same owner is unknown until the
   spike runs. Both copy variants are pre-authored: (a) confirmed (new address) → "rotates
   your server key"; (b) refuted (same address) → "resets the credential chain". Old-address
   float disposition → pre-cancel sweep dialog.

4. **L4 — Nuclear (irreversible):** Pre-destruction dialog shows the live remaining float
   balance and a **"Withdraw first"** offer. On confirm: the entire credential chain is
   hard-deleted (OIDC subject + all keys + JWT material); status becomes `nuked`
   (irreversible tombstone); the dashboard renders the "not provisioned" chip family with
   honest note. **Post-nuke floats are NOT withdrawable via Leash** — the withdraw operation
   is itself TEE-signed and the credential no longer exists. Copy: "Permanently destroys the
   signing credential. There is no recovery." Exit: provision a new agent.

Source: UX research §C; reintegration H-1..H-4; `research/x402-tab-leash-mechanics.md §5`.

**R-DASH-5 — Autonomy / cap adjustment.**
The owner can raise or lower the cap mid-cycle from the dashboard without restarting the
agent. The change takes effect on the next spend-cap check in the agent loop. The dashboard
shows the new cap immediately on save. Source: UX research §C (Brex "Allow limit increase
requests" toggle; spend limit config fields).

**R-DASH-6 — Mobile monitor (PWA).**
The mobile surface is a responsive Progressive Web App (proposed: Next.js `/mobile` route or
separate Vite build). It provides:
- **Monitor:** `ua.getPrimaryAssets()` USD total balance; the x402 payment feed (same receipt
  data as the web dashboard).
- **Notify:** Web Push notifications for Tier 2 and Tier 3 events.
- **Revoke:** All four revoke levels accessible as HTTP calls to `apps/web`; no private key
  or EIP-7702 authorization on-device.

No `signAuthorization`, `signMessage`, or private key ever exists on the mobile surface.
Source: `research/x402-tab-leash-mechanics.md §5` (phone = read + control client only);
DECISIONS.md (mobile = PWA, proposed pending Abu's confirm).

## Acceptance Criteria

**AC-TAB-1 — Invisible-crypto buyer flow.**
A buyer completes a payment by entering only an email address and a 6-digit OTP code. No
word "chain", "gas", "bridge", "Arbitrum", "EOA", "wallet address", or "sign" appears at
any point in the buyer-facing checkout UI from `<PayButton>` click to success screen.

**AC-TAB-2 — Cross-chain settlement on Arbitrum.**
A buyer whose primary assets are held ONLY on a chain other than Arbitrum (e.g., ETH on
Base, USDC on Polygon — zero Arbitrum balance) successfully pays a merchant, and the
merchant's address on Arbitrum One (`chainId: 42161`) shows an increased USDC (or configured
token) balance visible in `tokenChanges` of the `TransactionResult`.

**AC-TAB-3 — Optimistic success before block finality.**
The checkout modal displays an animated success state with `transactionId` immediately when
`ua.sendTransaction` returns — not after any polling interval or block-confirmation check.
The time between `sendTransaction` resolving and the success state appearing is zero
additional wait.

**AC-TAB-4 — Webhook receipt fires.**
After verified settlement, Tab dispatches the first POST inline to the merchant's configured
webhook URL. Each HTTP attempt waits at most 10 seconds for a response. The body contains
`transactionId` and `tokenChanges` that match the payment amount and the merchant's address
on Arbitrum. Settlement-to-first-attempt latency remains an internal target pending B-12 and
is not a merchant-facing SLA.

**AC-TAB-5 — 7702 mode is provable to judges.**
(a) The Particle UA is initialized with the nested `smartAccountOptions: { useEIP7702: true }`
form. (b) `sendTransaction` is called with the `authorizations` array (3rd argument) built
from `transaction.userOps`. (c) The transaction record at
`universalx.app/activity/details?id=${result.transactionId}` shows a successful cross-chain
transfer originating from the buyer's Magic-email EOA address.

**AC-TAB-6 — Magic email-OTP auth works end-to-end.**
A new buyer completes `loginWithEmailOTP` in the checkout modal without installing MetaMask,
without seeing a seed phrase, and without navigating away from the merchant page. The buyer
is in a wallet-provisioned state (EOA address available) on return from the OTP screen, with
no separate "your wallet is ready" screen shown.

**AC-LEASH-1 — Agent pays x402 resource autonomously via MCP proxy or fetch wrapper.**
The owner's external agent (Claude Code, Cursor, or equivalent) calls an x402-gated resource.
Leash intercepts the 402 via the MCP stdio proxy (primary) or the `@x402/fetch` wrapper
(secondary), pays from the correct pre-positioned float (routing by CAIP-2 network), receives
a 200 response, and a receipt row appears in the spend dashboard within the dashboard's
polling interval — with no code change in the agent and no human intervention per payment.

**AC-LEASH-2 — Spend cap is a hard block enforced by the hosted signer.**
When the agent's next payment would push cumulative spend over the owner-set cap, the Magic
Server Wallet (hosted signer) rejects the EIP-3009 signing request before any payment is
submitted. The Leash middleware receives the rejection, logs the blocked attempt, and emits a
Tier 3 (interrupt) notification. The block persists until the owner raises the cap or resets
the cycle from the dashboard. The agent cannot bypass this block by retrying.

**AC-LEASH-3 — Soft pause halts the agent.**
Triggering "soft pause" from the dashboard halts the agent loop. No x402 payment is made
after the pause action takes effect. The owner can resume the agent and payments proceed
normally on the next loop iteration.

**AC-LEASH-4 — Nuclear revoke blocks re-invocation.**
Triggering the L4 nuclear action: (1) presents the owner with a pre-destruction dialog
showing live remaining float balance and a "Withdraw first" offer; (2) on confirm, hard-deletes
the entire credential chain (OIDC subject + all keys + JWT material); (3) sets status to
`nuked` (irreversible tombstone); (4) renders the dashboard "not provisioned" chip family.
Any subsequent signing request is rejected — the credential no longer exists. Post-nuke floats
are NOT withdrawable via Leash (the withdraw path is itself TEE-signed). Re-provisioning
requires provisioning a new agent (new OIDC subject, new wallet, new address).
Source: reintegration H-3.

**AC-LEASH-5 — UA float rebalancing fires a real cross-chain op.**
When any pre-positioned float drops below the threshold, `apps/agent` issues a
`ua.createTransferTransaction` call that routes USDC from the unified treasury on one chain
to the agent's address on the target float chain. The resulting `transactionId` is visible at
`universalx.app/activity/details?id=...` and the float balance on the target chain increases.

**AC-DASH-1 — Real-time receipt feed accuracy.**
Every settled x402 payment appears in the dashboard feed within the polling interval, showing
amount (USD-denominated), resource URL, timestamp, and a `txHash` that links to a live block
explorer page for the settlement network.

**AC-DASH-2 — Spend bar reflects cumulative spend accurately.**
The spend bar displays a **settled segment** (solid fill, sum of all `settled` receipts in
the current cycle) plus an explicit **pending segment** (lighter fill or "incl. $X pending"
sub-line) for in-flight receipts. The cap gate checks `SUM(settled + pending)` so the 403
rejection threshold matches the number shown to the owner — a blocked payment is always
explainable by the displayed total. Blocked and failed receipts do not contribute to either
segment. The bar updates within the polling interval. Source: reintegration K-2, F-6.

**AC-DASH-3 — Notification tiers fire at the correct thresholds.**
(a) A routine payment (well below cap) does NOT trigger a Tier 2 or Tier 3 notification.
(b) A payment that pushes cumulative spend to or past 75% of cap triggers a Tier 2 (banner)
notification. (c) A payment that would push cumulative spend past 100% of cap triggers a
Tier 3 (interrupt) notification and the payment is blocked.

**AC-MOBILE-1 — PWA revoke reaches backend and halts agent.**
Triggering a revoke action from the mobile PWA results in the same agent halt as the web
dashboard within the HTTP round-trip latency. No EIP-7702 authorization, `signMessage`, or
private key is involved on the mobile device.

**AC-RAIL-1 (stretch) — Agent pays Tab checkout headlessly.**
When Tab exposes an x402 endpoint, the Leash agent pays it via `fetchWithPay(tabEndpoint)`
with no modal, no human input, and no popup. The merchant's webhook receives the same payload
structure (transactionId, tokenChanges) as a human payment.

## Constraints

1. **7702 signer must be an embedded / WaaS wallet or a raw server key.** EIP-7702 mode
   (`useEIP7702: true`) is only available with Magic, Privy, or Dynamic (the verified short
   list) or a raw server key. MetaMask / extension / JSON-RPC wallets cannot produce a Type-4
   `authorizationList` and are explicitly unsupported. Source:
   `wiki/sdks/particle-universal-accounts.md` (surface support, explicit `<Warning>` in
   eip7702-wallets.mdx).

2. **x402 cannot pay from the chain-abstracted unified balance directly.** The CDP
   facilitator checks `balanceOf(from)` on-chain on the exact network named in the 402 response;
   the UA's unified balance is invisible to it. Leash pre-positions USDC floats at the agent's
   EOA address on Base (`eip155:8453`, primary) and Arbitrum One (`eip155:42161`). Polygon
   (`eip155:137`) is excluded from v0 — UA SDK V2 removed Polygon rebalance support, making
   Polygon a money-in / no-money-out dead-end (see Constraint 14, R-LEASH-1). The Particle
   UA SDK is used **separately and asynchronously** to top up floats — the two flows must never
   be wired together inside the x402 payment loop. Source:
   `research/x402-tab-leash-mechanics.md §2` (✅ verified);
   `research/2026-07-02-x402-multichain-strategy.md §2–3`.

3. **EIP-3009 requires `ecrecover` — only a key-holding EOA.** Standard USDC v2 does NOT
   accept EIP-1271 smart-contract signatures (ERC-7598 is proposed, not shipped). A UA in
   Smart Account (4337) mode CANNOT sign x402. A UA in EIP-7702 mode CAN because UA address
   == owner EOA in that mode. Source: `research/x402-tab-leash-mechanics.md §2` (✅
   verified; coinbase/x402 issue #639 for 4337 support — open, not shipped).

4. **Third-party settlement is doc-proven but NOT yet run live (must-verify).** `createTransferTransaction(receiver=merchant)` to an external address on Arbitrum, funded from
   assets on a different chain, has been confirmed in the API docs but never executed in the
   cloned demos. A funded live spike with real Particle + Magic keys is a prerequisite for
   demo recording. Source: `research/x402-tab-leash-mechanics.md §4(a)` (⚠️ marked
   unverified live; action: funded live spike).

5. **Flagship eligibility rests on the HUMAN's UA doing the qualifying cross-chain op.**
   Whether an agent-held server key counts as "the user's EOA" for the flagship track is an
   open judging interpretation, not a technical question. The safest architecture: human's
   Magic-email EOA is the 7702 UA (satisfies the flagship requirement); the agent holds its
   own separate server key for x402. Source: `research/x402-tab-leash-mechanics.md §4(b)`
   (⚠️ unverified; action: ask organizers).

6. **No cryptographic session-key delegation on a UA in 7702 mode.** Particle session keys
   are Biconomy-v2.0.0-only and cannot attach to a UA in EIP-7702 mode. The Leash spend cap
   is enforced server-side by the hosted signer (Magic Server Wallet TEE), not by an on-chain
   delegation primitive or a Particle session key. Code must not call any session-key API on a
   UA. Source: `research/x402-tab-leash-mechanics.md §4`; `wiki/sdks/particle-universal-accounts.md`
   (gotchas: session keys Biconomy-v2.0.0-only, `aa/guides/keys.mdx:17`).

7. **Mobile signs nothing; revoke is a backend action.** The mobile PWA holds no private key
   and performs no EIP-7702 authorization, `signMessage`, or `signAuthorization`. Revoke is
   an HTTP POST to the `apps/web` backend. A phone-signed on-chain revoke would require an
   on-device 7702 signer, which is undocumented for UA and out of scope. Source:
   `research/x402-tab-leash-mechanics.md §5` (✅ verified).

8. **No custom Solidity contract in MVP.** No `packages/contracts` / Foundry is added unless
   a concrete, judge-convincing on-chain requirement emerges. The current evidence says app-layer
   policy is the correct and honest architecture. Source: DECISIONS.md (no custom contract —
   proposed pending confirm); `quality/2026-07-02-project-quality-profile.md` (open question 4).

9. **On-demand cross-chain top-up is too slow for the x402 hot path.** Particle UA
   `createTransferTransaction` end-to-end cross-chain latency is estimated at 10–90 seconds
   (L2→L2 fast path ~10–20s; Ethereum source chain ~60–90s). x402 `maxTimeoutSeconds` can be
   as low as 60 seconds (lower-bound production examples found; 300s is common). The EIP-3009
   `validBefore` clock cannot be paused during a bridge. Attempting on-demand bridging blocks
   every x402 call and will fail against tight-timeout resources. Pre-positioned floats with
   async background rebalancing is the only viable hot-path architecture. Source:
   `research/2026-07-02-x402-multichain-strategy.md §2` (verdict: "on-demand top-up breaks the
   x402 payment loop").

10. **CDP facilitator active network coverage.** Active for the CDP hosted facilitator: Base
   (`eip155:8453`), Polygon (`eip155:137`), Arbitrum One (`eip155:42161`), World (`eip155:480`),
   Solana Mainnet. Ethereum Mainnet, Optimism, and Avalanche are "reference only" — NOT active
   in the CDP facilitator. Source: `research/2026-07-02-x402-multichain-strategy.md §1`
   (network table); time-sensitive — re-check at build.

11. **Solana x402 is out of scope (v0).** x402 on Solana uses the `exact_svm` scheme,
    `@x402/svm`, and Ed25519 signing — a completely separate code path from the EVM/EIP-3009
    path. EIP-7702 is EVM-only. Solana hard-limits `maxTimeoutSeconds` at ~80–90 seconds.
    Solana does not add to the Particle UA Track or Arbitrum prize stack. Source:
    `research/2026-07-02-x402-multichain-strategy.md §5`.

12. **UA V2 migration in progress (time-sensitive).** The `UNIVERSAL_ACCOUNT_VERSION` import
    and account version must be verified at build time. Users may need to withdraw from old
    accounts before the demo. Source: `wiki/sdks/particle-universal-accounts.md` (gotchas).

13. **Stack (proposed, pending Abu's confirm on monorepo and mobile surface).** TypeScript
    monorepo: pnpm + Turborepo. Packages: `packages/sdk` (PayButton), `apps/web` (Next.js 16.2.6+
    App Router: checkout + intent API + webhook + Leash dashboard), `apps/agent` (MCP stdio
    proxy + hosted signer integration + UA float watcher), `apps/mobile` (PWA monitor).
    Libraries: viem, `@particle-network/universal-account-sdk@2.0.3`, `magic-sdk@33.9.0` /
    `@magic-sdk/admin` (embedded wallet + Server Wallets TEE), `@x402/{core,fetch,evm,mcp}@2.17.0`,
    `@modelcontextprotocol/sdk@^1.29.0`, `jose`, `web-push`, Drizzle + Postgres. **Dropped:**
    Vercel AI SDK, `@openrouter/ai-sdk-provider` (no LLM in Leash). Source: DECISIONS.md
    (stack — PROPOSED); `quality/2026-07-02-project-quality-profile.md`; reintegration
    §Integration Inventory (pinned package versions).

14. **UA SDK V2 init constraints (pinned: `@particle-network/universal-account-sdk@2.0.3`).**
    Three init-shape rules enforced in code review:
    - **Nested `smartAccountOptions` only.** `ownerAddress`, `useEIP7702`, `name`, and
      `version` belong inside `smartAccountOptions`. A flat top-level `ownerAddress` or
      `useEIP7702` is the old V1 quickstart shape and fails in V2.
    - **No `universalGas`.** The `tradeConfig: { universalGas: true }` option is removed in
      V2. Do not include it — it is either silently ignored or causes an error.
    - **No Polygon rebalance.** UA SDK V2 removed Polygon from supported rebalance networks.
      `createTransferTransaction` targeting Polygon is unsupported; Polygon is excluded from
      v0 floats accordingly.
    Source: reintegration §Integration Inventory; `@particle-network/universal-account-sdk@2.0.3`
    d.ts / CHANGELOG.

   owner sees each receipt appear in the feed; receives a Tier 2 banner at 75% cap; receives a
   Tier 3 interrupt when the cap is hit.

6. **Owner revokes (spectrum):** owner triggers each revocation level from the dashboard (soft
   pause → resume → freeze → cancel → nuclear); each level produces the correct agent behavior
   and the correct notification.

7. **UA float rebalancing demo:** with the Base float below threshold, the `apps/agent` watcher
   triggers a `ua.createTransferTransaction` top-up from the unified treasury to the agent's
   Base address; the resulting `transactionId` is confirmed on the universalx.app explorer; the
   dashboard shows the Base float balance increase.

8. **The qualifying human cross-chain op (flagship RAT):** a funded buyer EOA on a
   non-Arbitrum chain signs a Particle UA-7702 `createTransferTransaction` to a third-party
   merchant address on Arbitrum One; settlement is confirmed on Arbiscan; this constitutes the
   flagship track's required cross-chain value move.

## Open Questions

**OQ-1 — Is Tab-as-x402-resource (R-RAIL-1, R-RAIL-2) MVP or stretch?**
The "one rail, two payers" narrative is the most differentiated part of the submission, but
Tab exposing an x402 endpoint so the Leash agent can pay it headlessly is marked ⚠️ unverified
end-to-end (novel, no prior documented combination of Particle UA + x402 seller side). Source:
`research/x402-tab-leash-mechanics.md §3`. Decision: commit to MVP or defer to polish?

**OQ-2 — Does an agent-held server key satisfy "the user's EOA" for the flagship track?**
The Particle docs say server-side 7702 with a raw key is a first-class supported path, but
the flagship track language says "user's EOA." Whether judges accept an agent key as "the
user" is a judging-interpretation question. Resolution: ask hackathon organizers or Discord
before committing the agent-as-7702-UA narrative. The safer default: human's Magic EOA is
the 7702 UA; agent is additive. Source: `research/x402-tab-leash-mechanics.md §4(b)` (⚠️
unverified).

**OQ-3 — Which x402 test resource to demo Leash against?**
No specific x402-gated endpoint is identified yet. Options: (a) Tab exposes its own x402
endpoint (collapses with OQ-1); (b) a known public x402 test API; (c) a minimal demo x402
server the team runs. A decision is needed before the agent story (Story 3) can be built.

**OQ-4 — Exact settlement token on Arbitrum.**
Code now pins native Arbitrum USDC at `0xaf88d065e77c8cC2239327C5EDb3A432268e5831`,
but live Particle liquidity and end-to-end settlement for that asset remain unproven until
B-04. The question therefore stays open at the real-liquidity boundary rather than the
address-selection boundary. Source: `wiki/sdks/particle-universal-accounts.md` and
`research/x402-tab-leash-mechanics.md §3`.

**OQ-5 — Pending live spike: external-receiver settlement (gating for Tab demo).**
`createTransferTransaction(receiver=merchant)` to an external Arbitrum address, sourced
cross-chain, is doc-proven but not run live. This spike — a funded UA transferring to an
external address and confirming on Arbiscan — must pass before demo recording. No amount of
spec work replaces it. Source: `research/x402-tab-leash-mechanics.md §4(a)`.

**OQ-6 — Mobile: PWA or Expo React Native?**
DECISIONS.md marks mobile=PWA as PROPOSED pending Abu's confirm. PWA is faster to ship and
requires no signing (verified: mobile signs nothing); Expo adds native push and app-store
look but introduces setup overhead. Since mobile does no UA-7702 signing, both are
architecturally valid. Source: DECISIONS.md; `quality/2026-07-02-project-quality-profile.md`
(open question 1).

**OQ-7 — Magic $500 bounty: does the client embedded-wallet buyer flow satisfy the "embedded
wallet" requirement?**
The buyer path uses `magic-sdk` (`loginWithEmailOTP`) as the 7702 signer — this is the
embedded client wallet. The Magic docs say "the $500 requirement is literally embedded wallet."
This should qualify, but the bounty requirement says "most creative implementation" — plain
email OTP is table-stakes for qualifying, not for winning. Worth confirming with organizers if
the Leash agent's use of a different signing path (server key, not Magic Server Wallet) could
undercut the $500 claim. Source: `wiki/sdks/magic.md` (bounty scope section, open questions).
