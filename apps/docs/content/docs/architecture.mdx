Three trust boundaries, three keys:

- **`sk_…`** never leaves your server — it mints signed intents, so the
  browser can never change a price.
- **`pk_…`** is public — it scopes the SDK's checkout calls to your
  merchant account, and settlement reporting also demands the buyer's own
  identity token.
- **`agent_sk_…`** lives only in the `tab-mcp` proxy environment — the AI
  agent itself never sees a key, and the cap is enforced outside the model.

## Agent x402 payment

<Mermaid
  chart={`sequenceDiagram
  participant A as AI agent
  participant P as tab-mcp proxy
  participant R as x402 resource
  participant T as Tab API
  participant H as Hosted signer
  A->>P: tool call
  P->>R: request
  R-->>P: 402 Payment Required
  P->>T: authorize within cap (agent_sk_)
  T->>T: cap check — blocked receipts recorded too
  T->>H: sign EIP-3009 authorization
  H-->>T: signature (key never leaves)
  P->>R: retry with payment
  R-->>P: 200 OK
  P-->>A: result (agent never saw a key)
  T->>T: receipt: amount, network, tx hash, cap context`}
/>

The design rule everywhere: **money truth is recorded evidence** — settled
means an on-chain hash, blocked means a recorded reason, and the model
never holds the pen.
