## 1. Architecture the matrix assumes (so every row names one source of truth)

**Auth/tenancy:** browser runs Magic headless email OTP → DID token → `POST /api/auth/verify` → `@magic-sdk/admin` validates → find-or-create `merchants` row (email UNIQUE = "one account per merchant") → signed session cookie (jose JWT). No demo accounts, no persona jumps. Sign-out (added surface) = `magic.user.logout()` + cookie destroy.

**How a payment record comes to exist (the M4 feed's real source):**
- *Human path:* the `<PayButton>` (packages/sdk) fetches the merchant's intent endpoint, runs the buyer flow (buyer domain), and after `ua.sendTransaction` resolves it POSTs a settlement report to Tab: `POST /api/v1/checkout/report` with `{ publishableKey, buyerDidToken, transactionId, tokenChanges, intentUrl, amount }`. The Tab server **does not trust the browser**: it validates the buyer DID (Magic Admin), attributes the merchant via the publishable key, and **verifies settlement on-chain** (Arbitrum RPC via viem: tx receipt + USDC `Transfer` log to `merchants.receiving_address`; and/or Particle transaction lookup — see (unverified) note in row M4-10) before marking `settled` and enqueueing the webhook. "Settled" on the dashboard therefore always names a chain-verified fact, never a browser claim.
- *Agent path (one rail, two payers):* `POST /api/pay/x402` returns 402 with `accepts:[{scheme:"exact", network:"eip155:42161", asset: 0xaf88…5831, payTo: merchant.receiving_address}]`; on `X-PAYMENT`/`PAYMENT-SIGNATURE` it calls the CDP facilitator verify/settle, records the payment with `payer_type='agent'`, normalizes the facilitator receipt (`txHash → settlements.tx_hash`) into the same webhook payload, returns 200 + receipt.
- *Test mode:* payments made with `pk_test_`/`sk_test_` keys run the full real product code path but settlement is **simulated by the platform and labeled** — exactly what the decided banner copy already says ("Test payments are simulated and do not move real funds"). Records carry `env='test'`, `verification_method='simulated_test'`, per-row TEST badges, and webhooks carry `livemode:false`. Live mode is the judged proof path.

**Webhooks:** one endpoint per merchant per env. HMAC secret generated at save, shown once, stored **encrypted-at-rest (AES-GCM), not hashed** — it must be recoverable because Tab signs every delivery with it (deviation from the API-key hash-at-rest pattern, by necessity; API keys stay SHA-256-hashed). Signature: `X-Tab-Signature: t=<unix>,v1=HMAC-SHA256(secret, "<t>.<body>")` (Stripe scheme, as drawn in M6). Delivery worker retries ×3 with exponential backoff, persists `attempt`, `next_retry_at`, `response_time_ms`, `response_body` (first 500 chars), terminal `gave_up`.

**Money loop (answering "how do merchants withdraw?" before Abu asks):** Tab is **non-custodial for merchants**. Settlement lands directly at `merchants.receiving_address` on Arbitrum One — there is no Tab balance to withdraw from. Proposed default at signup: receiving address = the merchant's own Magic EOA (real, provisioned by their own login — zero config), **editable in Settings** to any address they control (edit affordance is an added deviation; prototype drew it read-only with no origin). If the Magic-EOA default is kept, the exit path is "the merchant controls that key via their Magic login"; whether Magic ships a usable send/export UI for it is (unverified) — the safer default is the editable field. **Abu decision D1.**

**Persistence:** Postgres (Neon or Supabase) + Drizzle ORM — **stack delta**: DECISIONS.md stack names no database; a real multi-tenant product requires one. **Abu decision D2.** Retry scheduling requires either a long-running Node worker or Vercel Cron/QStash — **Abu decision D3 (hosting)**.
