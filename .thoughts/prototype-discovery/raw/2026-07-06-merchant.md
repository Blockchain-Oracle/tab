## 0. Shared dashboard shell (appears in Keys / Quickstart / Transactions / Webhooks / Webhook Log / Settings frames)

**Structure**
- Left sidebar, 224px, `#F4F3EE`: Tab logo (blue `T` tile + "Tab" wordmark).
- Nav items (exact order): **Quickstart · Transactions · API keys · Webhooks · Settings** (5 items). Active item gets `#E9E7E0` pill + darker text.
- No nav item for the Webhook delivery log (M6 is a breadcrumbed sub-page of Webhooks: "Webhooks › Delivery log").
- No nav item for the demo page (matches resolved OQ-M7: M1 is a separate route).
- Sidebar bottom: `MODE` label + segmented TEST/LIVE control. TEST active = white pill, amber text `#96650F`; LIVE active = white pill, green text `#257A46`. Persistent on every dashboard frame (M7 requirement satisfied).
- Below mode: account block — avatar tile "A", **"Atlas Coffee Roasters"**, **"dev@atlascoffee.com"**. No dropdown, no sign-out, no account menu affordance anywhere.
- Top-of-content test banner (test mode only): "You are in test mode. Test payments are simulated and do not move real funds." — exact decided copy. Transactions frame uses the variant "Showing test payments only. Test payments are simulated and do not move real funds." (matches M4 TEST_MODE state copy).
- Fonts: Geist / Geist Mono (Newsreader only on canvas + demo page). Accent blue `#3B5BD9`, green `#257A46`, amber `#96650F`, red `#B4383D`.

**Data objects implied by shell**: session/tenant object `{ businessName, email, avatarInitial, mode: 'test'|'live' }`.

**Gaps**: no sign-out; no account switcher (fine — one account per merchant); mode toggle in LIVE→TEST direction has no design (decided: instant, no confirm — acceptable).

**OTP step**
- Back chevron + "Back" label (exits to email step).
- Mail icon, "Check your email", "Enter the code we sent to **dev@atlascoffee.com**".
- 6 OTP boxes, 44×52px; states per box: filled (char), active (blue border + caret), empty, error (red), disabled (grey). Mock code: **4 8 2 9 1 3**.
- OTP_AUTO_SUBMITTING: all boxes filled + "Verifying…" spinner row.
- Banners: wrong = "That code didn't match. Try again." (red); expired = "This code has expired. We've sent a new one." (amber); rate-limited = "Too many attempts. Please wait 5 minutes." (red, boxes disabled).
- "Didn't get it? **Resend code**" link (shown on otp/otp_wrong/otp_expired). NO post-resend ~30s countdown state designed.

**Redirect step**
- SUCCESS: green animated check draw, "Account created", "Taking you to your Quickstart…" (→ M2, matches map).

**States present**: IDLE, SUBMITTING, OTP_AWAITING_INPUT, OTP_AUTO_SUBMITTING, OTP_WRONG_CODE, OTP_EXPIRED, OTP_RATE_LIMITED, EMAIL_ALREADY_REGISTERED, SUCCESS, ERROR — all 10 from the surface map. **Nothing dropped.**

**CRITICAL DELTA (whole auth surface)**: the surface map / decisions say merchant auth = "standard Magic email-OTP (same SDK as buyer flow)" and the buyer-side decision is "Magic SDK renders email + OTP screens; Tab does NOT build a custom OTP screen" (R-TAB-4, AC-TAB-6 — buyer-scoped). The prototype renders a **fully custom Tab-branded email + OTP UI** for merchant auth. Implementing this as drawn requires Magic's headless/custom-UI email OTP mode rather than Magic's own modal (capability must be verified against Magic SDK docs — marked **(unverified)**). Alternative: accept Magic's whitelabel modal and treat this frame as visual aspiration only. Decision needed before build.

**Mocks**: `dev@atlascoffee.com`, code `482913`, "wait 5 minutes" value.

## 5. M2 — Quickstart (`_Tab Quickstart`)

**What it shows**: dashboard shell (Quickstart active), header "Quickstart" + "From install to your first settled payment. One action per step." + right-aligned progress label "**N of 6 steps complete**".

**Steps (numbered circles; done = green check, active = blue filled number, pending = outlined number; active step gets blue border card):**
1. **Install** — code: `npm install @tab/sdk` (matches resolved OQ-M1).
2. **Create your API key** — desc "Secret keys are shown once. You'll paste it into your server environment." CTA "Create API key" (when active). Code: `const tab = new Tab("<YOUR_API_KEY>");` → after KEY_CREATED auto-populates `const tab = new Tab("sk_test_••••••••a3f9");` (masked injection — matches map).
3. **Create your intent endpoint** — desc "Your server pins the amount and receiving address. Buyers can't modify it." Code block (verbatim):
   ```
   // app/api/tab/intent/route.ts
   export async function GET() {
     return Response.json({
       amount: "12.00",
       currency: "USD",
       receiver: "0x1a2b...cd9e",
       token: { chainId: 42161, address: USDC_ARBITRUM }
     });
   }
   ```
   (decimal-string amount matches resolved OQ-M6; shape matches R-TAB-2.)
4. **Add the pay button** — desc "One component. It handles auth, balance, confirmation, and success." Code: `<PayButton intentUrl="/api/tab/intent" onSuccess={(transactionId, tokenChanges) => showOrderConfirmation(transactionId)} />` (matches R-TAB-1 props).
5. **Configure your webhook** — desc "The webhook is your fulfillment signal — onSuccess alone is not reliable." CTA "Add webhook URL" (when active). Code: `POST https://api.atlascoffee.com/webhooks/tab` + `{ "transactionId": "0x8f2e...1b7d", "tokenChanges": [...] }`.
6. **Test it** — desc "Complete a $12.00 payment on your demo page. We'll detect it automatically." CTA "Open demo page" (when active).
7. **Go live** (appears only in LIVE_MIGRATION state) — desc "Real payments settle to your receiving address as USDC on Arbitrum One." CTA "Switch to Live mode". Amber note: "Reminder: swap sk_test_ for your live key in production. Test keys never move real funds."

- Every code block has a "Copy" chip.
- FIRST_TEST_PAYMENT milestone banner (green): "Test payment received. Your integration is working." + "**$12.00 from maya.chen@example.com · webhook delivered in 342ms**" + CTA "Switch to Live mode".
- LOADING: 4 shimmer step-card skeletons.
- State→done mapping in logic: empty=0, key_created=2, steps_progress=4, first_payment=6, live_migration=6.

**States present**: all 6 from map (LOADING, PRE_INTEGRATION_EMPTY, KEY_CREATED, STEP_COMPLETE, FIRST_TEST_PAYMENT, LIVE_MIGRATION).

**Deltas / gaps**
- Canvas note resolves the auto-detect designer's call: "Key creation and the first test payment are auto-detected; other steps progress manually." BUT no manual mark-as-done affordance is drawn on any step (no checkbox, no "Mark done" button). How steps 1/3/4 flip to done is undesigned. Note KEY_CREATED shows done=2 (steps 1 AND 2 done) — step 1 "Install" apparently completes as a side effect of key creation, which contradicts "manual" progression. Needs a product decision.
- Milestone banner reveals buyer email + webhook latency → data joins across payment + webhook delivery + buyer identity (see hidden requirements).
- Progress label caps at "6 of 6" even when the 7th "Go live" step is visible — minor copy logic quirk.

**Mocks**: `sk_test_••••••••a3f9`, `0x1a2b...cd9e`, `USDC_ARBITRUM` symbol, `0x8f2e...1b7d`, `https://api.atlascoffee.com/webhooks/tab`, "$12.00 from maya.chen@example.com", "342ms", progress counts.

## 13. Hidden requirements the design reveals

1. Magic email-OTP in **custom/headless UI mode** for merchant auth (custom OTP boxes, custom email form, resend, error banners) — capability unverified; must be checked against Magic SDK docs before committing to this visual.
2. **Buyer identity persistence**: every transaction must store payer type (Human/Agent) AND buyer email; merchant-facing exposure of buyer email is a privacy/product decision now baked into two screens.
3. **Quickstart progress engine**: per-tenant step state; auto-detection of key creation and first test payment; masked-key injection into code samples; live progress label; undesigned manual-completion affordance for the remaining steps.
4. **Go-Live gating service**: boolean checks for live-key-exists, webhook-configured-and-verified, test-payment-completed; behavior for unmet items undecided.
5. **Webhook verification concept** ("verified", "Listening") — likely realized via "Send test webhook" delivering a synthetic signed payload and recording success; needs definition.
6. **Signature contract**: `X-Tab-Signature: t=<unix-ts>,v1=<hmac-sha256-hex>` — timestamped Stripe-style scheme is now the drawn contract; webhook docs and verification code must match.
7. **Retry scheduler with exposed ETA**: "Next retry in 4 min", "Attempt N of 3", terminal "gave up" — backend must persist attempt history + next-retry-at per delivery.
8. **Manual "Resend delivery"** endpoint + audit implications (does a resend create a new delivery row?).
9. **Transactions Export** (CSV?) endpoint — or explicitly cut the button.
10. **Filtering + pagination APIs**: status filter, date-range filter, "Showing X of Y" counts, Previous/Next paging on transactions and deliveries.
11. **Test/Live data partitioning end-to-end**: env-scoped keys (`sk_test_`/`sk_live_` pre-provisioned live keys visible with "Never" used), env-scoped transactions/webhooks, banner/badge rendering driven by mode.
12. **Key metadata**: name (optional), type (secret/publishable), permissions (full/sending; publishable = "—"), created date, last-used tracking ("2 min ago", "Never") — last-used requires per-request key-usage stamping.
13. **Secret lifecycle coupling**: deleting the webhook endpoint discards its signing secret; regenerate invalidates old immediately (copy exists in map; flow undesigned).
14. **Publishable key provisioning path** — create modal mints secret keys only; `pk_` rows exist; something must create them (auto-provision at signup?).
15. **Relative + ISO dual timestamps** everywhere (title-attr tooltips) — consistent time formatting util.
16. **Copy-to-clipboard affordances** on: txid (row + detail), receiving address, full key, signing secret, webhook payload, code blocks.
17. **Webhook status derivation on transaction rows** must come from the same delivery records shown in M6 (mock inconsistency proves the risk of two sources of truth).
18. **"One account per merchant"** — email-unique tenant constraint, sign-up vs log-in disambiguation (EMAIL_ALREADY_REGISTERED exists; the inverse "no account, sign up instead" path on login is NOT designed).
19. **Session management**: RETURNING_SESSION silent redirect requires Magic session validity check on auth routes; sign-out flow entirely undesigned.
20. **First-run seeding ambiguity**: M3 EMPTY state exists, but default mock shows "Server key" + "Storefront" and Go-Live checklist expects a live key — decide what (if anything) is auto-provisioned at signup.
