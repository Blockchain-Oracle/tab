# EIP-7702

> Sources: [EIPS/eip-7702](https://eips.ethereum.org/EIPS/eip-7702), [7702collective.com](https://7702collective.com/), [Pectra mainnet announcement](https://blog.ethereum.org/2025/04/23/pectra-mainnet), [ZeroDev docs](https://docs.zerodev.app/). See [../../raw/sources.md](../../raw/sources.md).

## What it is (plain English, define jargon inline)
EIP-7702 ("Set Code for EOAs") lets a **normal wallet act like a smart wallet — in place, with the same address.**

- An **EOA** (Externally Owned Account) is a plain key-controlled wallet like MetaMask: an address derived from a private key, holding no code, able only to sign and send single transactions. "Dumb."
- A **smart account** is a wallet whose behavior is defined by a contract: it can batch many actions into one click, let someone else pay the gas (sponsorship), accept passkeys/session keys instead of a seed phrase, and add recovery. Historically you had to **deploy a brand-new contract account and migrate your funds to it** to get these.
- EIP-7702 removes the migration. The EOA signs an **authorization** saying "run this contract's code as if it were mine." From then on the address behaves like the smart account that code defines, **while keeping its original address, balance, history, and the original private key still working.**

So instead of "make a new smart wallet and move everything over," it is "upgrade the wallet you already have, reversibly." This is why the hackathon calls it the headline trick: one login can become a gasless, batched, agent-friendly smart account without the user ever creating a second address. [eip-7702; 7702collective.com]

## How it works (technically correct, concise)
EIP-7702 adds a new transaction type **`0x04`** (`SET_CODE_TX_TYPE`, a.k.a. the "type-4 / SET_CODE tx") that carries an **`authorization_list`**. [eip-7702]

Each entry in that list is an **authorization tuple** with six fields:
`[chain_id, address, nonce, y_parity, r, s]` — chain id, the **delegate contract address** whose code to adopt, the authorizing account's nonce, and the ECDSA signature (`y_parity, r, s`). [eip-7702]

- **Signing payload:** `msg = keccak(MAGIC || rlp([chain_id, address, nonce]))` with `MAGIC = 0x05`. The signer recovered from `(r,s,y_parity)` is the **authority** — the EOA being upgraded. A `chain_id` of `0` makes the authorization valid on any chain; a specific id pins it to one chain. [eip-7702]
- **What gets written:** the account's code is set to a 23-byte **delegation designator** `0xef0100 || address`. The `0xef` prefix is a reserved/banned opcode, so this value can never be normal contract bytecode — clients treat it as a pointer. [eip-7702]
- **Execution semantics:** when anything calls the EOA (`CALL`/`STATICCALL`/etc.) or the EOA itself executes, the EVM follows the pointer and runs the **delegate's code in the authority's own context** — i.e. with the EOA's storage, balance, and address (DELEGATECALL-like). The delegate contract is shared library code; per-user state lives at each EOA. [eip-7702]
- **Persistence, re-delegation, revocation:** the delegation is stored onchain and **persists across transactions** until changed. Re-delegate by signing a new authorization; **revoke** by delegating to the zero address `0x0000...0000`, which resets the account's code hash to the empty-code hash. The original private key keeps working the whole time. [eip-7702]
- **Gas:** `PER_AUTH_BASE_COST` = 12,500 gas per authorization; `PER_EMPTY_ACCOUNT_COST` = 25,000 (refunded if the account already exists); cold delegation resolution adds ~2,600. [eip-7702]
- **ERC-4337 link:** the `address` can point straight at existing **ERC-4337** smart-account code (4337 = the prior-gen account-abstraction standard that runs smart wallets via a "UserOperation" mempool + bundler, no protocol change). 7702 is not a replacement; it lets a raw EOA opt into that same 4337 wallet logic. The two compose. [eip-7702] → see [erc-4337-account-abstraction.md](erc-4337-account-abstraction.md)

### Security caveats (do not gloss over)
- **One signature = full account control.** Authorizing points all of the EOA's execution at the delegate; a malicious or buggy delegate (or a phishing signature) can sweep the account. The delegate must enforce its own auth. [eip-7702]
- **Delegates need replay protection** (e.g. a nonce) and should explicitly sign over `value`, `gas`, `target`, and `calldata`; otherwise actions can be replayed or redirected. [eip-7702]
- **`tx.origin == msg.sender` can break:** code now runs in an EOA's frame that isn't the top-level sender, weakening contracts that used that check (e.g. some sandwich/anti-bot guards). [eip-7702]
- **Relayer risk:** a third party submitting the tx can have its nonce invalidated or balance assumptions broken by the authorizing account. [eip-7702]
- **Storage-collision risk on re-delegation:** switching to a different delegate that lays out storage differently can corrupt state, since the storage stays at the EOA. (Mitigation = ERC-7201-style namespaced storage; *unverified which sponsor SDKs enforce this*.)

### Status
**Final** and **shipped.** EIP-7702 went live on Ethereum mainnet in the **Pectra** upgrade on **2025-05-07** (epoch 364032). It is active today; this is production tech, not a testnet proposal. [eip-7702; blog.ethereum.org/2025/04/23/pectra-mainnet]

## Why it matters for THIS hackathon
The hackathon's single thesis is **"make crypto invisible"** (see [overview](../hackathon/overview.md)). EIP-7702 is the cleanest lever for that:
- A non-crypto user logs in once with the wallet/email they already have — **no second address, no migration** — and immediately gets **gas sponsorship** (app pays fees), **batching** (approve+swap in one click), and **session keys / passkeys** (no seed-phrase prompts). That is exactly the "a normal person could use it without knowing crypto is underneath" bar the judges reward.
- It pairs directly with **chain abstraction** (one balance across chains) — the upgraded EOA is the account that a chain-abstraction layer drives. See [chain-abstraction.md](chain-abstraction.md).
- It is the enabling primitive for **agent wallets**: an AI agent can be granted a scoped session key on a user's existing EOA-turned-smart-account rather than custody of a fresh wallet. See [agent-wallets.md](agent-wallets.md).
- Strategically: building *on* 7702 aligns the project with the headline sponsor (Particle) and the General-Track / bonus sponsors (ZeroDev, Openfort, Magic), all of whom ship 7702 paths — letting one wedge stack multiple prizes (see [tracks-and-prizes.md](../hackathon/tracks-and-prizes.md)).
